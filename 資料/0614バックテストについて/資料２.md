## 1. 使い方

## 結論

バックテストで最初に作るべきものは、戦略探索画面でも高機能なシミュレータでもない。
最初に作るべきものは、**嘘をつかない最小会計エンジン**である。

## 最低合格ライン

どのプロジェクトでも、まず以下を満たす。

```text
1. 入力データschemaが固定されている
2. timestamp意味論が固定されている
3. signal生成時点とfill時点が分離されている
4. Order / Fill / Position / Portfolioが別contractになっている
5. fee / spread / slippage / funding or financing を分けて記録する
6. PnLにslippageを二重控除しない
7. no-lookahead testがある
8. portfolio accounting testがある
9. metrics.json / fills / orders / equity_curve がartifactとして残る
10. runtime dataに依存しないdeterministic fixture testが通る
```

## 最初に切り捨てるもの

MVPでは以下を入れない。

```text
- live発注
- wallet / signing / nonce / cloid lifecycle
- 複数取引所adapter
- Web UI
- 高度なoptimization UI
- ML自動探索
- full L2 replay
- queue position full model
- paper/live calibration
- MT5 / CFD / Perp / 株式の同時抽象化
```

## 判断原則

```text
目的:
  勝てることを証明するのではなく、壊れる条件を早く見つける。

設計:
  中核は小さく、会計は厳密に、約定モデルは段階的に。

OSS:
  参考・検算・探索補助として使う。正本会計を置き換えない。

レポート:
  良さそうに見える説明より、落とした理由と不確実性を残す。
```

## 2. バックテストシステム概要

## バックテストの定義

バックテストとは、過去の時系列データ上で、戦略がその時点で利用可能だった情報だけを使って判断し、注文・約定・会計・コスト・制約を再現し、戦略候補の採用可否を評価する仕組みである。

## 誤った定義

```text
過去データで利益が出るかを見る。
```

これは弱い。正しくは以下。

```text
その利益が、データ品質・時間制約・コスト・約定・会計・過適合を考慮しても残るかを見る。
```

## システム構成

```text
Data Source
  ↓
Raw Data Store
  ↓
Normalizer / Schema Validator
  ↓
Feature Builder
  ↓
Signal Generator
  ↓
Order Generator
  ↓
Fill Model
  ↓
Portfolio / Accounting
  ↓
Metrics
  ↓
Validation / WFO / Bootstrap / Null Tests
  ↓
Report / Artifacts
```

## バックテスト方式

| 方式 | 特徴 | 向く用途 | 主なリスク |
|---|---|---|---|
| Vectorized | 配列で高速計算 | 大量パラメータ探索 | 約定・状態遷移が粗い |
| Bar-based event loop | 足ごとに処理 | 初期MVP | 同一barのhigh/low使用でlookaheadしやすい |
| Tick / trade-based | trade tickで処理 | 短期戦略 | データ量・欠損・順序問題 |
| L1 quote-based | bid/askで処理 | spread/slippage検証 | depth不足 |
| L2 replay | 板深さで処理 | execution realism | 実装・データ・計算が重い |
| Event-driven broker simulation | 注文状態を再現 | paper/live近似 | 複雑化しやすい |

## 段階的ロードマップ

```text
Stage 1:
  deterministic bar fixture
  long-only / single-symbol
  market-like fill
  fee + spread/slippage

Stage 2:
  real normalized data
  tradability gate
  funding/financing
  regime breakdown

Stage 3:
  multi-symbol
  position sizing
  WFO / bootstrap / null tests

Stage 4:
  quote-aware / trade-aware fill
  capacity and liquidity stress

Stage 5:
  L2 replay / latency / paper-live calibration
```

## バックテストが答えられる問い

- コスト控除後に利益が残るか
- どの市場状態で壊れるか
- 過剰最適化ではないか
- 取引数が十分か
- 手数料・spread・funding・slippageのどれに食われるか
- no-lookaheadが守られているか
- 実行制約込みでentry可能だったか

## 答えられない問い

- liveで完全に同じ約定が得られるか
- 取引所APIが同じように応答するか
- WebSocket欠落・REST/WS不一致が起きないか
- 実注文履歴が信用できるか

これらはLEMのような執行監査システムで検証する。

## 3. データ契約と時刻意味論

## 目的

バックテストの多くの事故は、戦略ロジックではなくデータ契約とtimestampの曖昧さから起きる。
したがって、最初に入力データの意味を固定する。

## 最小データ契約

Bar / quote系MVPでは以下を持つ。

```text
ts
symbol
open
high
low
close
volume
mid_price
best_bid
best_ask
spread_bps
is_tradable
block_reasons
source_confidence
venue_quality_score
```

perpでは追加。

```text
mark_price
oracle_price
index_price
funding_rate
funding_interval
open_interest
fee_mode
taker_fee_bps
maker_fee_bps
market_status
session_type
```

CFD / FXでは追加。

```text
bid
ask
spread
commission
swap_long
swap_short
contract_size
point_value
digits
session_open
session_close
margin_rate
```

株式では追加。

```text
corporate_actions
split_adjustment
dividend
borrow_fee
shortable_flag
trading_calendar
halt_status
```

## timestampの種類

```text
event_ts:
  市場イベントが発生した時刻。

available_ts:
  戦略がその情報を利用可能になった時刻。

decision_ts:
  戦略が判断した時刻。

order_ts:
  注文意図を作った時刻。

fill_ts:
  約定したと仮定する時刻。

local_recv_ts:
  collectorが受信した時刻。
```

最重要ルール:

```text
feature.available_ts <= decision_ts
fill_ts > decision_ts
```

## no-lookaheadルール

Bar closeでsignalを作るなら、同じbarの高値・安値でentry fillしてはいけない。

推奨:

```text
signal:
  bar[t] close後に生成

entry:
  bar[t+1] のexecutable priceでfill

exit:
  exit signalがbar[t] close後なら、bar[t+1]以降でfill
```

## データ品質検査

最低限検査する。

```text
- timestamp逆行
- 重複timestamp
- symbol欠落
- price <= 0
- bid >= ask
- spread異常
- volume < 0
- stale data
- missing rows
- block_reasons型不一致
- timezone混在
```

## data manifest

各runで以下を出す。

```json
{
  "dataset_id": "trade_xyz_quotes_2026_05",
  "input_path": "data/normalized/quotes.parquet",
  "schema_hash": "...",
  "row_count": 123456,
  "symbols": ["SP500"],
  "time_start": "2026-05-01T00:00:00Z",
  "time_end": "2026-05-31T23:59:00Z",
  "timezone": "UTC",
  "normalizer_version": "...",
  "quality_status": "pass"
}
```

## 4. Signal / Order / Fill / Portfolio契約

## なぜ契約を分けるか

戦略が直接PnLを作る設計は壊れやすい。
Signal、Order、Fill、Position、Portfolioを分けることで、どこでバグったかを特定できる。

## Signal

Signalは売買判断であり、まだ注文ではない。

```json
{
  "signal_id": "sig_001",
  "decision_ts": "2026-05-31T10:00:00Z",
  "symbol": "SP500",
  "side": "buy",
  "strength": 0.82,
  "strategy_id": "breakout_20_10",
  "features_ref": "features.parquet#row=100",
  "reason": "20_period_high_breakout"
}
```

## Order

Orderはバックテスト内の発注意図。live adapterのOrderIntentと混ぜない。

```json
{
  "order_id": "ord_001",
  "signal_id": "sig_001",
  "ts": "2026-05-31T11:00:00Z",
  "symbol": "SP500",
  "side": "buy",
  "order_type": "market",
  "qty": "1.0",
  "limit_price": null,
  "reduce_only": false,
  "source": "backtest"
}
```

## Fill

Fillは約定モデルの結果。fill_price_sourceを必ず残す。

```json
{
  "fill_id": "fill_001",
  "order_id": "ord_001",
  "ts": "2026-05-31T11:00:00Z",
  "symbol": "SP500",
  "side": "buy",
  "qty": "1.0",
  "fill_price": "5250.25",
  "fee": "4.725225",
  "liquidity_flag": "taker",
  "fill_price_source": "best_ask",
  "slippage_impact": "1.25"
}
```

## Position

Positionは保有状態。

```json
{
  "symbol": "SP500",
  "qty": "1.0",
  "avg_price": "5250.25",
  "realized_pnl": "0",
  "unrealized_pnl": "12.5",
  "fees_paid": "4.725225",
  "funding_pnl": "0"
}
```

## Portfolio

Portfolioはcash/equity/positionsを持つ。

```json
{
  "ts": "2026-05-31T11:00:00Z",
  "cash": "9745.024775",
  "equity": "10007.774775",
  "positions": [{"symbol": "SP500", "qty": "1.0"}],
  "realized_pnl": "0",
  "unrealized_pnl": "12.5",
  "fees_paid": "4.725225"
}
```

## 不変条件

```text
flat start:
  position.qty == 0

long entry:
  qty increases
  avg_price == first_fill_price
  realized_pnl unchanged
  fees_paid increases

long exit:
  qty returns to 0
  realized_pnl reflects trade economics
  unrealized_pnl returns to 0

blocked entry:
  no order/fill/position mutation

no duplicate fill:
  same fill_id/order_id cannot be counted twice
```

## 5. 約定・コスト・会計

## 原則

PnLは会計から計算する。見かけの売買差益へ後から雑にコストを引く設計は避ける。

## 約定モデルの段階

| 段階 | 入力 | fill価格 | 用途 |
|---|---|---|---|
| Bar next open | OHLCV | next open | 最小MVP |
| Bar conservative | OHLCV | buy=high寄り / sell=low寄り | 安全側スクリーニング |
| L1 quote | bid/ask | buy=ask / sell=bid | spread反映 |
| Trade tick | trades | next trade price | 短期検証 |
| L2 sweep | order book | depthを食う | 大きな注文・薄い板 |
| Queue model | L2/L3 | maker fill確率 | maker/limit評価 |

## market-like fillの推奨

Perp / CFD / FXのMVPでは以下でよい。

```text
long entry:
  exec_buy_price -> best_ask -> mid_price + spread/2 -> close + conservative_slippage

long exit:
  exec_sell_price -> best_bid -> mid_price - spread/2 -> close - conservative_slippage
```

必ず `fill_price_source` を残す。

## コストの種類

```text
fee:
  取引手数料。maker/takerを分ける。

spread:
  bid/ask差。market-like fillならfill_priceに含まれる。

slippage:
  reference priceとの差。fill_priceに含めた場合、PnLから二重控除しない。

funding:
  perpの定期支払い。position size * price * rate。

financing / swap:
  CFD / FXの保有コスト。

borrow:
  株式shortの借株コスト。

market impact:
  自分の注文サイズが市場価格へ与える影響。MVPでは無理に入れない。
```

## slippage二重控除禁止

悪い例:

```text
fill_price = ask + slippage
pnl = sell - buy - fee - slippage
```

これはslippageを二重に引く可能性がある。

良い例:

```text
fill_priceに実行価格を反映する。
pnlはfill_price同士で会計する。
slippage_impactはreference priceとの差として別metricsに出す。
```

## fee model

feeはhardcodeしない。

```text
優先順位:
  1. row上の解決済み fee_bps
  2. config上のfee_mode
  3. market profile default
  4. 不明ならentry禁止
```

## funding / financing

funding_rateの単位を推測しない。

```text
必要情報:
  rate単位
  適用間隔
  適用時刻
  long/short符号
  使用価格 oracle/mark/mid
```

不明な場合:

```text
funding_cost = 0
funding_status = unknown_or_null
metricsにはfunding列を残す
```

## cash accounting例

Long entry:

```text
cash -= qty * fill_price
cash -= fee
position.qty += qty
position.avg_price = fill_price
```

Long exit:

```text
cash += qty * fill_price
cash -= fee
realized_pnl += (exit_price - avg_price) * qty - entry_fee_alloc - exit_fee - funding
position.qty -= qty
```

Derivativeでは証拠金とnotional会計を分ける必要がある。MVPではnotional-based PnLを明示し、margin liquidationまでは入れない。

## 6. 検証・WFO・Bootstrap・過適合

## なぜ単純なbacktestだけでは足りないか

単一期間で良い成績が出ても、以下を除外できない。

```text
- 期間依存
- パラメータ過適合
- データ漏洩
- 取引数不足
- コスト仮定の甘さ
- 一部レジームだけの偶然
- 探索回数を隠したp-hacking
```

## 基本分割

```text
train:
  パラメータや特徴量を決める期間

validation:
  候補選別期間

test / holdout:
  最終確認期間。繰り返し使わない。
```

## Walk-forward Analysis

WFOは、時系列に沿って train → test をずらしながら評価する。

```text
fold 1: train [0..90d]  test [91..104d]
fold 2: train [14..104d] test [105..118d]
fold 3: train [28..118d] test [119..132d]
```

見るもの:

```text
- fold別net pnl
- median fold
- worst fold
- positive fold ratio
- fold別trade count
- fold別cost drag
- fold別regime breakdown
```

## Embargo / Purge

特徴量やlabelが未来期間を参照する場合、train/test境界に隙間を置く。

```text
train_end = 2026-05-01 00:00
embargo = 1 day
test_start = 2026-05-02 00:00
```

## Bootstrap

tradesやreturnsを再標本化し、成績の不確実性を測る。

```text
bootstrap_samples = 1000
sample_unit = trade or day or block
outputs:
  net_return_ci
  max_drawdown_ci
  profit_factor_ci
  probability_net_positive
```

時系列相関が強い場合、単純trade bootstrapよりblock bootstrapを検討する。

## Null Tests / Negative Controls

```text
random signal:
  ランダムentryで同程度に勝てるなら戦略の価値が弱い。

label shuffle:
  labelをシャッフルしても同程度なら特徴量が怪しい。

time shift:
  signalを未来/過去へずらして成績が大きく変わるか確認。

reverse signal:
  売買反転で同じように勝つならコスト・会計を疑う。

asset shuffle:
  symbol入れ替えで同じなら市場固有性が弱い。
```

## Overfitting Guard

最低限、以下を記録する。

```text
trials_count
parameter_count
feature_count
conditions_count
search_space_size
selected_candidate_rank
all_trials_registry
```

## 合格条件の例

```text
- net pnl > 0 after fee/spread/funding
- trade_count >= min_trade_count
- median fold > 0
- worst fold drawdown below threshold
- no single month contributes > 50% profit
- null tests do not match strategy performance
- no proxy/approx metric used as sole PASS reason
```

## 7. Regime / 特徴量 / ML検証

## レジーム判断の位置づけ

レジーム判断は、最初から高度な分類器にしない。
MVPでは、**entry gate + tag + breakdown** として作る。

## RegimeTagger最小仕様

入力:

```text
market row
  ts
  symbol
  market_status
  session_type
  spread_bps
  depth_10bps_usd
  funding_rate
  source_confidence
  venue_quality_score
  is_tradable
  block_reasons
```

出力:

```json
{
  "ts": "2026-05-31T10:00:00Z",
  "symbol": "SP500",
  "tags": ["tradable", "regular_session", "thin_depth"],
  "entry_allowed": true,
  "blocked": false,
  "reasons": []
}
```

## 最初のタグ

```text
tradable
blocked
closed_or_maintenance
regular_session
wide_spread
thin_depth
high_funding_abs
low_source_confidence
low_venue_quality
high_premium
high_oi_usage
```

## 閾値はconfig化する

```yaml
regime:
  wide_spread_bps: 20
  thin_depth_10bps_usd: 100000
  high_abs_funding_rate: 0.0005
  min_source_confidence: 0.8
  min_venue_quality_score: 0.8
```

## no-lookahead

レジーム判定にもlookaheadがある。

禁止:

```text
次barのvolatilityで現在barのregimeを決める
当日終値まで見て日中regimeを決める
future drawdownを使ってstress regimeを付ける
```

許可:

```text
decision_ts時点で利用可能なspread/depth/funding/source_confidenceだけでtagを付ける
rolling特徴量はavailable_tsを持つ
```

## Regime別metrics

```json
{
  "regime_breakdown": {
    "wide_spread": {
      "trade_count": 12,
      "net_pnl": -53.2,
      "avg_slippage_bps": 18.1
    },
    "regular_session": {
      "trade_count": 88,
      "net_pnl": 421.5,
      "avg_slippage_bps": 4.4
    }
  }
}
```

## PyBrokerの使いどころ

PyBrokerは、RegimeTaggerや特徴量が効いているかをwalk-forward・bootstrapで検証する補助として有用。ただし正本会計やTrade[XYZ]固有制約の中核にはしない。

使う用途:

```text
- regime featureの有効性確認
- model-based signalのwalk-forward評価
- bootstrap metrics
- parameter / feature比較
```

使わない用途:

```text
- fee/funding accounting正本
- block_reasons / venue_quality gate正本
- live order lifecycle再現
- LEM代替
```

## 特徴量検証の出力

```text
feature_importance.json
walkforward_metrics.json
bootstrap_metrics.json
regime_feature_report.md
```

ただし最終採用判定は自前engine + RobustLab Gateで行う。

## 指標定義

```text
total_return:
  コスト控除前またはgross基準。必ず定義を明記。

net_return_after_cost:
  fee/funding/spread反映後の資産増減。

max_drawdown:
  equity curve peakからの最大下落。

profit_factor:
  gross_profit / abs(gross_loss)。

exposure_time:
  positionを持っていた時間比率。

turnover:
  取引notional合計 / 平均equity。

cost_drag_bps:
  gross成績とnet成績の差をbps化。

slippage_impact:
  reference priceとfill priceの差。PnLから二重控除しない。
```

## 9. テスト方針と受け入れ条件

## テスト哲学

バックテストは「それっぽい結果が出る」だけでは不十分。
以下をテストで固定する。

```text
- データschema
- no-lookahead
- fill price selection
- fee / funding calculation
- portfolio accounting
- blocked entry
- artifact output
- deterministic replay
```

## 必須unit tests

```text
test_schema_required_columns:
  必須列欠落でfailする。

test_fee_unknown_blocks_entry:
  feeが不明ならentryできない。

test_block_reasons_block_entry:
  block_reasons非空なら新規entryしない。

test_is_tradable_false_blocks_entry:
  is_tradable=falseで新規entryしない。

test_fill_price_priority_buy:
  exec_buy_price -> best_ask -> mid/spread の順に使う。

test_fill_price_priority_sell:
  exec_sell_price -> best_bid -> mid/spread の順に使う。

test_long_entry_accounting:
  qty, avg_price, cash, feesが正しく更新される。

test_long_exit_accounting:
  realized_pnlとflat化が正しい。

test_no_lookahead_future_price_change:
  未来barを変えても過去のdecision/fillが変わらない。

test_slippage_not_double_counted:
  fill_priceに含まれるslippageをPnLから二重控除しない。

test_artifacts_written:
  metrics/orders/fills/equity_curveが出る。
```

## Property tests候補

```text
fee_bpsを増やすとnet pnlは改善しない
slippageを悪化させるとnet pnlは改善しない
同一seed/同一inputなら結果が一致する
同じfillを二重投入しても二重計上されない
```

## 10. OSSバックテストツール利用方針

## 結論

OSSは使える。ただし、正本会計・Trade[XYZ]固有制約・no-lookahead・artifact contractの中核は自前で持つ。
OSSは以下に限定する。

```text
- 設計学習
- 高速探索
- metrics検算
- walk-forward / bootstrap補助
- L2 / latency検証の将来サンドボックス
```

## 推奨マトリクス

| OSS | 用途 | 位置づけ | 中核採用 |
|---|---|---|---|
| PyBroker | ML特徴量、walk-forward、bootstrap | research harness | しない |
| vectorbt | 高速パラメータ探索 | research helper | しない |
| QuantStats | metrics / tear sheet検算 | report helper | しない |
| NautilusTrader | event-driven設計参考 | future reference | v0.1ではしない |
| HftBacktest | L2/latency/queue検証 | future sandbox | v0.1ではしない |
| backtesting.py | API参考・簡易比較 | reference only | ライセンス注意 |
| Backtrader / LEAN | 設計参考 | reference only | しない |

## PyBroker

向く用途:

```text
- RegimeTaggerや特徴量が効くかのwalk-forward検証
- ML風モデルのtrain/test評価
- bootstrap metrics
- indicator/model cache
- parallel computation
```

制約:

```text
- Apache 2.0 with Commons Clause / non-commercial 表記に注意
- 本体依存にしない
- optional dependencyまたはexperiments隔離
```

## vectorbt

向く用途:

```text
- 大量パラメータ探索
- signalだけの高速比較
- 複数timeframeの粗い比較
```

制約:

```text
- Apache 2.0 with Commons Clause
- 正本accountingにはしない
```

## QuantStats

向く用途:

```text
- max drawdown / Sharpe / Sortino / Calmarなどの検算
- report UX参考
```

制約:

```text
- Trade[XYZ]固有のfee/funding/slippage/blocked reasonは自前metricsが正本
```

## NautilusTrader

学ぶべき点:

```text
- event-driven architecture
- data granularityとbook_typeの整合
- funding settlement
- order/position/accountingの境界
- repeated runs / reset semantics
```

今やらない理由:

```text
- v0.1に対して重い
- live/event-drivenへ引っ張られる
```

## HftBacktest

向く用途:

```text
- L2/L3 replay
- feed/order latency
- queue position
- future fill model validation
```

今やらない理由:

```text
- HFT/MM向け要素が強い
- 現プロジェクト制約ではHFT/MM/arbを避ける
- v0.1はbar/L1相当で十分
```

## 14. 付録B: 失敗モード

## データ系

```text
- 欠損をforward fillして取引可能に見せる
- timestampのtimezoneが混在する
- adjusted priceとraw priceを混ぜる
- delisted銘柄を除外してsurvivorship biasになる
- symbol mappingを暗黙変換する
- stale dataを最新データ扱いする
```

## 時間系

```text
- signal barとfill barが同じ
- high/lowを見てからそのbarでentryする
- future fundingを現在decisionに使う
- その日の終値で日中regimeを決める
```

## コスト系

```text
- feeをhardcodeする
- maker/takerを区別しない
- funding単位を推測する
- slippageを二重控除する
- spreadを無視する
- tax/financing/borrowを対象市場で必要なのに無視する
```

## 約定系

```text
- すべてcloseで約定する
- 薄い板でも大口が同一価格で約定する
- limit orderが触れただけで必ず約定する
- partial fillを無視する
- trading halt中にentryする
```

## 会計系

```text
- realized/unrealizedが混ざる
- feeをentry/exitで二重または未計上
- short会計がlongと同じ式になる
- derivative notionalとcash marginを混同する
```

## 検証系

```text
- holdoutを何度も見る
- trial失敗を記録しない
- パラメータ探索回数を隠す
- null testをしない
- warning-onlyをPASS扱いにする
```

## OSS系

```text
- OSS engineの出力を正本にする
- OSSのデフォルトfee/slippageを理解せず使う
- ライセンス確認なしに組み込む
- optional dependencyが本体必須になる
```
