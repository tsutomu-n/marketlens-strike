結論: **完成形は「Trade[XYZ]専用の純粋バックテストエンジン」です。**
売買Bot本体でも、live発注システムでも、RobustLab/LEM本体でもありません。既存Repo `marketlens-strike` の中に、**Trade[XYZ]のSP500などを対象に、戦略がコスト後に本当に成り立つかを検証する基盤**を追加するものです。

# 1. 完成形はなにか

完成後のものは、Repo内の新しいバックテスト基盤です。

```txt
marketlens-strike
  └─ src/sis/backtest/
      ├─ engine/
      │   ├─ order.py
      │   ├─ fill.py
      │   ├─ portfolio.py
      │   ├─ runner.py
      │   └─ metrics.py
      │
      └─ trade_xyz/
          ├─ schema.py
          └─ cost_model.py
```

役割はこれです。

```txt

  ↓

  ↓

  ↓

  ↓

  ↓
RobustLabへ渡せるcandidate_resultを作る
```

現行Repoにはすでに Trade[XYZ] のread-onlyデータ取得、Strategy Research Lab、paper-only preview、venue quality gate、既存の簡易backtest surfaceがあります。ただし、既存の `uv run sis build-backtest` や `src/sis/backtest/bridge.py` はStrategy Lab / paper preview寄りであり、今回作る「独立した会計・約定・コスト検証エンジン」とは分ける方針です。

# 4. 最初の対象戦略

v0.1では、最初から複雑な戦略を入れません。

固定するMVPはこれです。

```txt
symbol:
  SP500

side:
  long-only

position:
  single-symbol only

entry:
  20期間高値ブレイク

exit:
  10期間安値割れ

order:
  market-like entry / market-like exit

fill:
  exec_buy_price / exec_sell_price 優先
  なければ best_ask / best_bid
  さらに欠ける場合は mid_price + spread_bps から保守推定

cost:
  fee + spread/slippage + nullable funding

filter:
  is_tradable=false はentry禁止
  block_reasons非空はentry禁止
  discovery bound / OI cap はentry gateのみ
```

このMVPは、資料上でも「SP500 / 1h相当 / long-only / single-symbol / market-like fill」として固定されています。

# 6. 何がわかるのか

このバックテストでわかることは、主に次です。

## 戦略がコスト後に残るか

```txt
net_return_after_cost
total_return
fee_impact
funding_impact
slippage_impact
cost_drag_bps
```

「見かけ上勝っているが、手数料・spread・funding後に死ぬ」戦略を落とせます。

## 損失耐性

```txt
max_drawdown
worst_trade_pnl
median_trade_pnl
profit_factor
win_rate
```

単に利益が出たかではなく、どの程度のDDを食らうかが見えます。

## 取引の質

```txt
trade_count
turnover
exposure_time
blocked_reason_counts
source_confidence_gate_pass_rate
venue_quality_gate_pass_rate
```

過剰売買、少なすぎる取引数、取引不可状態でのentryなどを検出できます。

## コストの内訳

```txt
fee
funding
slippage
spread由来の悪化
```

「戦略が悪い」のか「コストに食われている」のかを分けて見られます。

## Backtestがズルをしていないか

```txt
no-lookahead
次bar約定
future bar priceを使っていないか
signal rowとfill rowの分離
```

資料では、signalはbar closeで生成し、entryは同じbarではなく次のexecutable row/barで行う方針が明記されています。

# 7. 逆に、何はわからないのか

このv0.1では、次はわかりません。

```txt
liveで本当に同じ価格で約定するか
L2板を食ったときの完全なsweep-depth
queue position
partial fillの精密挙動
Hyperliquid userFillsとの照合
実注文のREST/WS不一致
本番のclock skew
MT5 / IC Markets / CFDでの成績
```

これらはLEMや将来のevent-driven / L2 replay / CFD別プロジェクトの範囲です。

このbacktest v0.1の役割は、まずこれです。

```txt
粗いが嘘の少ない初期スクリーニング。
```

# 10. 要するに

```txt
完成形:
  Trade[XYZ]専用の純粋バックテストエンジン。

使い方:
  Trade[XYZ]のデータと戦略シグナルを入れて、
  注文・約定・手数料・funding・slippage・Portfolio会計を再現し、
  metricsとartifactを出す。

わかること:
  その戦略が、Trade[XYZ]のコスト・取引制約・約定前提込みで、
  初期候補として残す価値があるか。

わからないこと:
  liveで本当に同じように約定するか。
  それはLEMの範囲。
```

このツールは、最終的にこういう位置づけです。

```txt
Strategy案
  ↓
Trade[XYZ] Backtest Engine
  ↓
コスト後に残る候補だけ抽出
  ↓
RobustLabで研究品質を検査
  ↓
paper / tiny-live
  ↓
LEMで実執行を監査
```
