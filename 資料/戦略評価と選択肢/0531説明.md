結論。
完成形は **「Trade[XYZ]専用の研究・検証用バックテスト基盤」** です。
これは売買Botそのものではなく、**ある戦略がTrade[XYZ]上で本当に期待値を持つか、手数料・スリッページ・funding・session・discovery bounds込みで検証するための装置**です。

いま作ったPR-1は、その完成形のうち **会計エンジンの土台** だけです。
つまり、現時点でわかるのは「ポジション、cash、fee、PnLを嘘なく計算する骨格ができた」というところまでです。

# 1. 完成形は何か

完成形は、ざっくり言うとこれです。

```text
Trade[XYZ]の過去データを読み込む
  ↓
戦略ロジックを走らせる
  ↓
注文・約定・手数料・funding・slippageを再現する
  ↓
ポジションと資産残高を更新する
  ↓
最終的に、戦略が勝てるか / どこで死ぬか / どの条件が効くかを出す
```

最終的な構成はこうなります。

```text
Backtest System
  ├─ Data Loader
  │    └─ Trade[XYZ]の過去データを読む
  │
  ├─ Market Schema
  │    └─ mark/oracle/bid/ask/funding/fee/session/boundsを正規化
  │
  ├─ Strategy Runner
  │    └─ 戦略がentry/exit/add/skipを判断
  │
  ├─ Fill Model
  │    └─ その注文がどの価格で約定した扱いになるかを決める
  │
  ├─ Cost Model
  │    └─ maker/taker fee、funding、slippageを計算
  │
  ├─ Portfolio Engine
  │    └─ cash、position、realized/unrealized PnLを更新
  │
  ├─ Risk / Gate
  │    └─ discovery bounds、OI cap、tradable flag、sessionで取引可否判定
  │
  └─ Report
       └─ 成績、破産率、MFE/MAE、forward return、損益分布を出す
```

最終成果物は、単なる「勝率」ではなく、こういうファイル群になります。

```text
backtest_run.json
orders.parquet
fills.parquet
trades.parquet
equity_curve.parquet
positions.parquet
metrics.json
data_manifest.json
backtest_report.md
```

# 3. 何がわかるのか

このバックテスト基盤でわかることは、主に5つです。

## 3-1. その戦略に期待値があるか

たとえば、

```text
手数料前は勝っている
でも手数料込みだと負ける
```

または、

```text
通常fee modeだと負ける
growth fee modeなら勝てる可能性がある
```

のようなことがわかります。

これは重要です。
Trade[XYZ]ではfee mode、spread、funding、sessionが効くため、単純な価格チャート上の勝ち負けでは判断できません。

---

## 3-2. どの条件で勝っているか

たとえば、最終的にはこういう分析ができます。

```text
CASH_SESSIONでは勝つ
OVERNIGHTでは負ける

discovery boundから遠い時は勝つ
upper bound接近時のlongは負ける

fundingが偏っている時だけ勝つ
spreadが広い時は期待値が消える
```

つまり、「戦略が勝つ」ではなく、**どの地合い・どの条件でだけ勝つか**が見えるようになります。

---

## 3-3. 負けの原因がわかる

負けたときに、原因を分解できます。

```text
シグナルが悪い
手数料で負けている
スリッページで負けている
fundingで削られている
session選別が悪い
discovery bounds付近で踏み増している
entryは良いがexitが悪い
```

この分解ができないと、戦略改善が勘になります。

---

## 3-4. 攻撃的複利を入れてよいか判断できる

最終的には、こういう比較ができます。

```text
固定ロット
  vs
equity percentage sizing
  vs
aggressive compound
  vs
pyramidあり
  vs
pyramidなし
```

これにより、

```text
複利を入れるとリターンは増えるが破産率も上がる
pyramidは一部sessionでは有効だが、bounds接近時は危険
initial cash 1000 USDでは最小注文制約に引っかかる銘柄がある
```

のような判断が可能になります。

つまり、攻撃的複利を「勢い」で入れるのではなく、**破産率・最大DD・期待値の比較で入れるか判断できる**ようになります。

---

## 3-5. live / paperに進める価値があるか判断できる

最終的な役割はこれです。

```text
この戦略は実弾化する価値があるか？
```

判断材料はこうです。

```text
期待値がプラスか
fee/slippage/funding込みで勝っているか
drawdownが許容範囲か
session別に頑健か
パラメータを少し変えても壊れないか
データ期間を変えても勝つか
破産率は許容できるか
```

これが通って初めて、paper tradingやmicro-liveへ進むべきです。

# 4. 今作ったPR-1は完成形のどこか

PR-1はここです。

```text
Backtest System
  ├─ Data Loader
  ├─ Market Schema
  ├─ Strategy Runner
  ├─ Fill Model
  ├─ Cost Model
  ├─ Portfolio Engine  ← 今ここ
```

つまり、**バックテスト中にcash、position、PnLを更新する中核会計部分**です。

今のPR-1でできること：

```text

equityを計算する
partial exitを処理する
blocked entryでは状態を変えない
```

今のPR-1でまだできないこと：

```text
Trade[XYZ]データを読む
戦略を走らせる
entry/exitシグナルを作る
約定価格を推定する
fee modelをconfigから読む
fundingを時系列で反映する
discovery boundsを判定する
バックテストレポートを出す
複利・pyramidを検証する
```

なので、PR-1は完成形ではなく、**完成形の土台**です。

```text

bounds近辺で死んでいるか
複利を入れると破産率がどう変わるか
```

# 7. 最終的に「わかること」と「わからないこと」

## わかること

```text
過去データ上で期待値があるか
手数料込みで勝っているか
スリッページ込みで勝っているか
funding込みで勝っているか
session別に勝っているか
bounds付近で負けているか
複利でリターンと破産率がどう変わるか
live化する価値があるか
```

## わからないこと

```text
未来でも勝つか
実際のlive約定が完全に同じになるか
突発的なoracle異常に耐えるか
API障害時に必ず逃げられるか
低流動時の本当の約定品質
```

だから完成形でも、これは **勝利保証装置ではなく、実弾化する価値を判定するための検証装置**です。
