結論：**システムトレードのバックテストは、「過去データで利益が出たか」を見る作業ではなく、戦略仮説が“手数料・スリッページ・約定不能・データ漏洩・相場局面の変化”に耐えるかを検査する工程**です。

## 1. バックテストとは何か

バックテストは、過去の市場データに対して、あらかじめ決めた売買ルールを機械的に適用し、以下を検証する作業です。

```text
入力データ
  ↓
特徴量生成
  ↓
売買シグナル生成
  ↓
注文・約定シミュレーション
  ↓
損益計算
  ↓
リスク評価
  ↓
採用 / 修正 / 破棄 / paper観測
```

重要なのは、バックテストは **未来の利益を証明しない** ことです。
バックテストが示すのは、せいぜい次のことです。

```text
この戦略は、過去のこのデータ、この仮定、このコスト、この約定モデルでは、
どの程度よく見えるか。
```

したがって、バックテストの結論は常に条件付きです。

---

## 2. 良いバックテストと悪いバックテスト

### 悪いバックテスト

```text
過去データで右肩上がりだった
Sharpeが高かった
勝率が高かった
最大DDが低かった
だから儲かる
```

これは危険です。

理由は、以下を無視している可能性が高いからです。

```text
手数料
スリッページ
スプレッド
約定不能
partial fill
funding
look-ahead bias
survivorship bias
overfitting
parameter fitting
銘柄選定の未来情報
流動性低下
運用サイズ
paper/live差分
```

### 良いバックテスト

```text
なぜ利益が出る仮説なのか
その仮説はどのデータから来たのか
未来情報を使っていないか
手数料・スリッページ後も残るか
約定できるか
相場局面を変えても残るか
パラメータを少し変えても壊れないか
paperで再現できるか
```

ここまで見るのが、システムトレードのバックテストです。

---

## 3. バックテストの階層

バックテストは1種類ではありません。段階があります。

## レベル0：動作確認バックテスト

目的は「コードが動くか」です。

```text
シグナルが出る
ポジションが建つ
損益が計算される
artifactが出る
```

これは必要ですが、戦略の優位性はほぼ分かりません。

## レベル1：シグナルバックテスト

目的は「売買ルール自体に筋があるか」です。

```text
entry
exit
side
holding period
stop loss
take profit
position sizing
```

この段階では、約定はまだ粗くてもよいです。

vectorbtのような高速なシグナル駆動バックテストはこの層に向いています。`Portfolio.from_signals` は long / short entry、exit、方向衝突、反対シグナルの扱い、stop loss / take profit などを扱えますが、同一tick・同一symbolで複雑な複数注文を扱う場合は、より細かい注文関数側の制御が必要になります。([VectorBT][1])

## レベル2：コスト込みバックテスト

ここから実務に近づきます。

最低限、以下を入れます。

```text
取引手数料
スプレッド
スリッページ
funding
借入コスト
税・資金効率
```

あなたの前提では、perpのtaker feeを **0.04% = 4bps** と見るので、往復takerだけでこうなります。

```text
entry fee  4bps
exit fee   4bps
合計       8bps
```

つまり、1回の売買で少なくとも **8bps** 以上の期待値が必要です。
ただし実際にはこれだけでは足りません。

```text
必要期待値 =
  8bps
  + entry slippage
  + exit slippage
  + funding buffer
  + safety margin
```

例：

```text
entry fee        4bps
exit fee         4bps
entry slippage   5bps
exit slippage    5bps
funding buffer   3bps
safety margin    5bps
---------------------
必要edge         26bps
```

この前提なら、平均26bps未満のトレードは、見た目の勝率が高くても危険です。

## レベル3：約定現実性バックテスト

低流動性銘柄ではここが本体です。

見るべきものは利益率だけではありません。

```text
注文が刺さるか
成行でどれだけ滑るか
板が薄い時にpartial fillになるか
no-fill率はどれくらいか
スプレッド拡大時にブロックできるか
depthが足りるか
latencyで価格が逃げるか
```

NautilusTraderのドキュメントでも、L2/L3 order book dataを使う場合は実際の板レベルに対して注文をfillし、サイズを各価格レベルで順にマッチさせることで、より現実的なスリッページと板厚の影響を扱う設計が説明されています。([NautilusTrader][2])

逆に、bar dataだけで約定を再現すると危険です。barのtimestampや処理順が間違うと、未来の高値・安値を知っていたような約定になり、look-ahead biasになります。NautilusTraderも、bar timestampの扱いを誤ると非現実的なバックテストになると警告しています。([NautilusTrader][2])

## レベル4：頑健性バックテスト

単一期間で良いだけでは不十分です。

見るべきもの：

```text
walk-forward
purged walk-forward
out-of-sample
bootstrap
block bootstrap
regime split
rolling stability
cost stress
slippage stress
benchmark relative
```

ここでは「一番良い成績」を探すのではなく、**どの条件で壊れるか**を探します。

## レベル5：paper observation

バックテストの次はpaperです。

paperで見るべきものは、最初から利益ではありません。

```text
バックテスト通りにシグナルが出るか
想定時刻に注文できるか
no-fill率はBTと近いか
スプレッド悪化時に止まるか
funding反映が合うか
paperの約定想定とBTがどれだけズレるか
```

バックテストは「仮想世界」。
paperは「実データ時系列に近い観測」。
liveはさらに別物です。

---

## 4. バックテストで見るべき指標

見るべき指標は、利益率だけではありません。

### 成績系

```text
total return
net return after cost
profit factor
win rate
average win
average loss
expectancy per trade
Sharpe-like ratio
Sortino-like ratio
max drawdown
recovery factor
```

ただし、勝率は単独では弱いです。

例：

```text
勝率80%
平均利益 +0.2%
平均損失 -1.2%
```

この戦略は普通に悪いです。

重要なのは期待値です。

```text
期待値 =
  勝率 × 平均利益
  - 敗率 × 平均損失
  - コスト
```

### 実務系

```text
trade count
turnover
fee paid
slippage paid
funding pnl
average spread bps
depth at entry
no-fill rate
partial-fill rate
blocked signal count
time in market
max concurrent positions
```

低流動性perpなら、実務系指標の方が重要なことも多いです。

### 頑健性系

```text
walk-forward positive rate
worst era return
best era / worst era gap
rolling max drawdown
regime別成績
cost stress survival
parameter sensitivity
benchmark relative return
```

## 5. バックテストで最も危ない罠

## 罠1：Look-ahead bias

未来の情報を使ってしまうことです。

例：

```text
当日終値でシグナルを判断し、
同じ当日終値で約定したことにする
```

これは実際にはできません。
終値が確定した時点では、その終値で入れないからです。

修正：

```text
シグナル判定: t時点の確定情報
約定: t+1以降
```

また、barのhigh/lowを使ったstop/take profit判定も注意が必要です。bar内の順序が分からない場合、stopとtake profitのどちらが先に到達したかは確定できません。

## 罠2：Survivorship bias

今も残っている銘柄だけで過去を検証することです。

cryptoなら特に危険です。

```text
消えた銘柄
上場廃止銘柄
流動性が死んだ銘柄
取引停止銘柄
```

これらを除外すると、過去成績が過大評価されます。

## 罠3：Parameter overfitting

過去に一番合うパラメータを探しすぎることです。

例：

```text
RSI 14では負ける
RSI 13なら勝つ
RSI 12.5ならもっと勝つ
```

このような改善は、未来では消えやすいです。

対策：

```text
parameter sweepの範囲を事前に決める
最良値ではなく安定領域を見る
out-of-sampleで確認する
変更回数を記録する
```

## 罠4：コスト軽視

特にperpでは致命的です。

```text
手数料
funding
spread
slippage
market impact
```

を少し甘くするだけで、戦略が別物になります。

## 罠5：約定前提が甘い

低流動性銘柄では、「シグナルが出た価格で全部約定」はまず疑うべきです。

必要なのは、

```text
best bid / ask
depth
spread
partial fill
no-fill
latency
```

です。

## 10. 実務上の結論

システムトレードのバックテストは、次の順序で使うのが正しいです。

```text
1. 戦略仮説を作る
2. 未来情報なしのデータを用意する
3. シグナルを生成する
4. まず軽いbacktestで粗く落とす
5. 手数料・funding・slippageを入れる
6. 約定現実性を入れる
7. walk-forward / stress / regimeで壊す
8. Review Packetにまとめる
9. 人間がレビューする
10. 承認済みだけpaperへ送る
```

あなたのプロジェクトで最も大事なのは、バックテストエンジンをさらに増やすことではありません。

**既存バックテストの結果を、戦略ごとに読みやすいReview Markdownへ変換し、paperへ進めるかを判断できるようにすること**です。
