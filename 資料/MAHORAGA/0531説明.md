結論：**完成形は「Trade[XYZ]専用の、実注文を出さない純粋バックテスト基盤」です。**
既存のStrategy Labやpaper-only評価とは別に、Repo内へ **注文・約定・ポジション・損益・手数料・funding・slippage・見送り理由を台帳として残す検証システム** を作る計画です。

## 完成形はなにか

完成後、Repoには大きくこの構造が追加されます。

```text
src/sis/backtest/
  engine/
    order.py        # 注文意図
    fill.py         # 約定
    portfolio.py    # ポジション・資金・PnL
    ledger.py       # 台帳
    runner.py       # バックテスト実行
    metrics.py      # 成績計算
    artifacts.py    # 出力保存

  trade_xyz/
    schema.py        # Trade[XYZ]用BT入力行
    input_adapter.py # quotes/registry/fee config からBT行へ変換
    fee_model.py     # 手数料解決
    cost_model.py    # コスト内訳
    fill_model.py    # best_bid/best_askによる約定価格
    funding_model.py # funding損益
    gate.py          # entry/exit可否判定
    strategies.py    # 最初はsnapshot breakout

src/sis/commands/
  backtest.py        # 最終的なCLI
```

最初の完成対象は、**SP500のTrade[XYZ] perpを対象にした最小バックテスト**です。

```text
対象:
  Trade[XYZ] SP500

戦略:
  SnapshotBreakoutStrategy

entry:
  過去20行のmark_price高値を上抜けたら、次rowでlong entry

exit:
  過去10行のmark_price安値を下抜けたら、次rowでexit

約定:
  long entry = 次rowのbest_ask
  long exit  = 次rowのbest_bid

記録:
  order / fill / position / equity / fee / funding / slippage / blocked reason
```

重要なのは、これは**儲かる戦略の完成形**ではありません。
完成形は、**戦略を検証できる土台**です。

## どう使うのか

使い方は3段階です。

### 1. データを用意する

入力は、Trade[XYZ]から取得・正規化されたquoteデータです。

```text
入力データ:
  data/normalized/quotes.parquet

補助データ:
  data/registry/trade_xyz_instrument_registry.json
  configs/fee_model.trade_xyz.yaml
```

バックテスト側では、これをそのまま使うのではなく、`TradeXyzBacktestRow` に変換します。

変換時に次を解決します。

```text
- price_event_ts_ms
- fee_mode
- taker_fee_bps
- maker_fee_bps
- funding_interval
- discovery_bound_bps
- max_leverage
- is_tradable
- block_reasons
```

特に今回のTrade[XYZ]データでは `oracle_ts_ms` がnullなので、既存risk gateをそのまま使うと約定が0件になります。
新しいbacktestでは、

```text
price_event_ts_ms = oracle_ts_ms ?? source_ts_ms ?? recv_ts_ms
```

として扱います。

### 2. バックテストを実行する

最終的にはCLIでこう使う想定です。

```bash
uv run sis backtest-trade-xyz-run \
  --quotes data/normalized/quotes.parquet \
  --registry data/registry/trade_xyz_instrument_registry.json \
  --fee-model configs/fee_model.trade_xyz.yaml \
  --symbol SP500 \
  --strategy snapshot-breakout \
  --initial-cash-usd 3000 \
  --notional-usd 1000 \
  --out-dir data/backtest/runs
```

最初の実装中は、CLIより先にpytestで実行します。

```bash
uv run pytest -q tests/backtest/test_trade_xyz_runner_breakout.py
uv run pytest -q tests/backtest/test_trade_xyz_no_lookahead.py
uv run pytest -q tests/backtest/test_portfolio_accounting.py
```

つまり、最初は「画面から使うツール」ではなく、**テストで意味論を固定した検証エンジン**として作ります。
CLIは最後に付けます。

### 3. 出力を見る

実行後、こういう成果物が出ます。

```text
data/backtest/runs/<run_id>/
  manifest.json
  config.json
  input_manifest.json
  orders.parquet
  fills.parquet
  positions.parquet
  funding_events.parquet
  equity_curve.parquet
  trades.parquet
  blocked_entries.parquet
  metrics.json
  report.md
```

見る順番はこれです。

```text

   人間向けの概要

2. metrics.json
   成績指標

3. trades.parquet
   個別トレード結果

4. fills.parquet

5. blocked_entries.parquet
   なぜ取引しなかったか

   資金曲線

7. funding_events.parquet
   fundingがどれだけ効いたか
```

## 何がわかるのか

このシステムでわかることは、主に5つです。

### 1. その戦略が、Trade[XYZ]の実コスト込みで成立するか

単に価格が上がった/下がっただけではなく、以下を差し引いて評価します。

```text
- taker fee
- maker fee
- spread crossing
- slippage
- funding
- entry不可条件
- exit時の実行価格
```

これにより、

```text
価格上は勝っているが、feeとspreadで負けている
```

というパターンを検出できます。

これは非常に重要です。
特にTrade[XYZ]のperpでは、単純にmark_priceで売買したつもりになると成績を過大評価します。

### 2. どの条件で取引がブロックされたか

`blocked_entries.parquet` に、entryできなかった理由を残します。

例：

```text
BLOCK_FEE_UNRESOLVED
BLOCK_NO_EXECUTABLE_BUY_PRICE
BLOCK_SPREAD_TOO_WIDE
BLOCK_DEPTH_TOO_THIN
BLOCK_DISCOVERY_BOUND_EDGE
BLOCK_MARKET_NOT_OPEN
BLOCK_ROW_NOT_TRADABLE
```

これにより、

```text
戦略が悪いのか
データが足りないのか
市場が閉じているのか
feeが未解決なのか
板が薄いのか
```

を分けて判断できます。

既存のStrategy Authoring backtestでは、Trade[XYZ]データが `BLOCK_ORACLE_TIMESTAMP_MISSING` で約定0件になっていました。
新しいpure BTでは、こうした「バックテスト側の仕様不一致による0件」を切り分けられます。

### 3. fee / funding / slippage のどれが損益を壊しているか

`metrics.json` や `trades.parquet` で、損益を分解できます。

```text
gross_pnl_usd
fees_paid_usd
funding_pnl_usd
slippage_cost_usd
net_pnl_usd
```

これで、たとえば次の判断ができます。

```text
戦略の方向性は合っている
しかしfeeで負けている
→ 取引頻度を下げるべき

grossでは勝っている
fundingで負けている
→ 保有時間を短くするか、funding-awareにする

spreadで負けている
→ SP500以外の個別株perpはまだ対象外にする
```

### 4. no-lookaheadが守られているか

新しいバックテストでは、ここを明示的にテストします。

```text
signal row i で判断
entry fill は row i+1
```

つまり、同じrowの価格を見て、同じrowで都合よく約定した扱いにはしません。

これにより、

```text
未来の価格を見てから過去で売買したことになっている
```

というバックテストの典型的な誤謬を防ぎます。

ここは収益性より重要です。
no-lookaheadが壊れているバックテストは、成績が良くても意味がありません。

### 5. paper / micro live に進める候補かどうか

最終的に、このpure BTは次の判断材料になります。

```text
この戦略はpaperへ昇格してよいか
この銘柄は対象に入れてよいか
この取引頻度は高すぎないか
fee/funding込みで期待値があるか
実行不能rowが多すぎないか
板が薄すぎないか
```

つまり、完成後の役割は、

```text
アイデア
  ↓
pure backtest
  ↓
paper
  ↓
micro live
  ↓
本番検討
```

のうち、**paperへ上げる前の検査工程**です。

## 何はわからないのか

逆に、この完成形だけでは以下はわかりません。

```text
- 実発注時の完全な約定品質
- exchange障害時の挙動
- wallet/signingまわりの安全性
- live order reconciliation
- 大きなサイズでのmarket impact
- 長期間の統計的な有意性
- 他銘柄・他市場でも通用するか
- MT5 / IC Markets CFDで通用するか
```

特にMVPはSP500中心なので、最初から次のような結論は出せません。

```text
Trade[XYZ]全体で勝てる
個別株perpでも勝てる
NASDAQやCFDでも同じ戦略が使える
本番運用してよい
```

完成形が示すのは、あくまで

```text
Trade[XYZ] SP500のこのデータ範囲で、
この実行モデルとコストモデルなら、
この戦略はどう振る舞ったか
```

です。

## 主案

主案は、**これを既存Repo内のTrade[XYZ]専用pure backtest layerとして完成させる**ことです。

```text
既存:
  Strategy Lab / paper-only fixed horizon評価

新規:
  Trade[XYZ] pure backtest
  order/fill/position/equity ledgerあり
  fee/funding/slippage分離
  no-lookahead保証あり
```

これが一番安全です。

## 代替案

既存 `bridge.py` を拡張してバックテストっぽくする方法もあります。
ただし、これは避けた方がよいです。

理由は、`bridge.py` はすでにpaper-only評価やStrategy Authoring用途を持っていて、そこに本格的な会計台帳を混ぜると責務が壊れるからです。

別Repoにする方法もありますが、今のRepoにはTrade[XYZ] collector、normalizer、registry、fee configがすでにあるので、別Repo化すると型やデータの乖離が起きやすいです。

## 使った後の判断例

完成後、たとえばこう判断できます。

### 例1：成績が良い場合

```text
net_pnl_usd > 0
max_drawdownが許容内
fees_paid_usdが小さい
blocked_entriesが少ない
no-lookahead testが通る
```

この場合：

```text
SP500 paper運用候補に昇格
```

### 例2：grossでは勝つがnetで負ける場合

```text
gross_pnl_usd > 0
net_pnl_usd < 0
fees_paid_usdが大きい
```

この場合：

```text
戦略の方向性はあるが、取引頻度が高すぎる
→ lookbackを長くする
→ entry条件を厳しくする
→ maker前提はまだ使わない
```

### 例3：blocked entryが多い場合

```text
blocked_entry_countが多い
BLOCK_FEE_UNRESOLVEDが多い
BLOCK_NO_EXECUTABLE_BUY_PRICEが多い
```

この場合：

```text
戦略以前にデータ/fee/fill仕様が未整備
→ 実運用検討は不可
```

### 例4：個別株perpでspread負けする場合

```text
SP500は成立
NVDA/AAPL/AMDではspreadやdepthで悪化
```

この場合：

```text
MVP対象をSP500/XYZ100に限定
個別株perpは後回し
```
