# 1. 完成形はなに？

完成形は、既存Repo内に追加される **Trade[XYZ]専用の純粋バックテスト基盤** です。

イメージはこれです。

```text
Trade[XYZ]の過去データ
  ↓
データ検証
  ↓
売買ルールを適用
  ↓
約定をシミュレーション
  ↓
手数料・スリッページ・Fundingを反映
  ↓
損益・DD・取引回数・コスト影響を集計
  ↓
レポート出力
```

最初のv0.1では、対象を広げません。

```text
対象:
  Trade[XYZ]のみ
  Perpetualsのみ
  まずSP500
  まず1h相当
  まずlong-onlyのブレイクアウト戦略

対象外:
  MT5
  IC Markets
  CFD
  他取引所
  live発注
  自動売買
  L2板再生
  高頻度取引
```

つまり完成形は、**「戦略を本番投入してよいか判断する前段の検証装置」** です。

---

# 2. どう使うの？

完成後は、だいたい次の流れで使います。

## 手順1：Trade[XYZ]のデータを用意する

必要なデータは、最低限こういうものです。

```text
timestamp
symbol
open / high / low / close
mark_price
oracle_price
bid / ask
volume
funding_rate
is_tradable
block_reasons
open_interest
open_interest_cap
discovery_bound
fee情報
```

生データをそのまま信じるのではなく、バックテスト用schemaに合わせて検証します。

---

## 手順2：設定ファイルを書く

例：

```yaml
symbol: SP500
timeframe: 1h
initial_cash: 1000
strategy:
  type: breakout
  entry_lookback: 20
  exit_lookback: 10
direction: long_only

execution:
  fill_model: next_bar_market_like
  no_lookahead: true

cost:
  fee_mode: trade_xyz
  include_spread: true
  include_slippage: true
  include_funding: true

gates:
  require_is_tradable: true
  block_on_discovery_bound: true
  block_on_oi_cap: true
```

---

## 手順3：バックテストを実行する

完成後のCLIイメージはこうです。

```bash
uv run sis backtest-trade-xyz \
  --config configs/backtest_trade_xyz_v0_1.final.example.yaml
```

最初はCLI公開を後回しにして、Python APIから使う設計でもよいです。

```python
from sis.backtest.trade_xyz.runner import run_backtest

result = run_backtest(config_path="configs/backtest_trade_xyz_v0_1.final.example.yaml")
```

---

## 手順4：出力を見る

出力されるものは、だいたいこうです。

```text
reports/backtest_trade_xyz/<run_id>/
  summary.md
  trades.csv
  equity_curve.csv
  metrics.json
  cost_breakdown.csv
  skipped_entries.csv
  config_resolved.yaml
  manifest.json
```

# 3. 何がわかるの？

主に、次がわかります。

## 1. その戦略は過去データで利益が出たか

```text
最終損益
勝率
平均損益
取引回数
最大ドローダウン
損益曲線
```

ただし、単に「利益が出たか」だけではなく、**どの条件で利益が削られたか** を見るのが重要です。

---

## 2. 手数料込みで勝っているか

Trade[XYZ]では手数料を雑に固定すると危険なので、fee modelを入れます。

わかること：

```text
gross pnl:
  コスト前の損益

fee:
  手数料

net pnl:
  手数料後の損益

fee impact:
  手数料でどれだけ利益が削られたか
```

例：

```text
コスト前 +3.2%
手数料 -1.1%
スリッページ -0.8%
Funding -0.2%
最終 +1.1%
```

こう見えるようになります。

---

## 3. スプレッド・スリッページで負けていないか

低流動性銘柄では、シグナルが正しくても約定で負けることがあります。

わかること：

```text
bid/askを考慮すると利益が残るか
スリッページを入れると赤字になるか
低流動性時に取引しすぎていないか
```

---

## 4. Fundingを入れると結果が変わるか

PerpではFundingが効きます。

わかること：

```text
Fundingを入れない場合の損益
Fundingを入れた場合の損益
Fundingが利益を押し上げたか
Fundingが利益を削ったか
```

v0.1では、funding_rateがない場合は無理に推測しません。
`None` なら `funding impact = 0` として扱い、レポートに「Funding未評価」と出すべきです。

---

## 5. 取引してはいけないタイミングを避けられたか

Trade[XYZ]固有の条件をentry gateとして見ます。

```text
is_tradable = false
block_reasonsあり
discovery bound違反
OI cap到達
session外
market_status不明
fee不明
```

わかること：

```text
本来入ってはいけないところで入っていないか
何件のentryがブロックされたか
どの理由でentryが拒否されたか
```

---

## 6. 未来情報を使っていないか

これはかなり重要です。

v0.1では、**no-lookahead** をエンジン契約にします。

つまり、

```text
あるbarでシグナル発生
  ↓
同じbarの価格では約定しない
  ↓
次のexecutable row/barで約定する
```

これにより、バックテストが実力以上に良く見える事故を防ぎます。

---

## 7. 戦略を次に進める価値があるか

最終的に知りたいのはこれです。

```text
この戦略は研究継続する価値があるか？
コスト込みで期待値が残るか？
取引回数は十分か？
DDは許容できるか？
entry gateでほとんど弾かれていないか？
Fundingやslippageを入れると壊れないか？
```

---

# 4. 完成後にできること・できないこと

## できること

```text
Trade[XYZ]データでバックテストする
手数料込み損益を見る
スリッページ込み損益を見る
Funding影響を見る
entry gateで取引不可条件を避ける
no-lookaheadで検証する
取引履歴をCSVで見る
損益曲線を見る
metrics.jsonを見る
summary.mdで結果を読む
SP500のサンプル戦略を検証する
```

## できないこと

```text
本番注文を出す
リアルタイム自動売買する
勝てる戦略を自動生成する
MT5 / IC Markets CFDを検証する
L2板を完全再生する
約定順序をtick単位で完全再現する
HFT戦略を検証する
RegimeGuardの高度検証まで自動で行う
Live Execution Monitorと本番約定比較する
```

v0.1は、あくまで **純粋バックテストの土台** です。

# 5. この計画の価値

このバックテスト基盤ができると、次の判断ができます。

```text
1. Trade[XYZ]で戦略研究を続ける価値があるか
2. SP500のような対象で最低限の優位性が残るか
3. 手数料・Funding・slippageを入れると壊れるか
4. entry gateを入れると取引機会が消えすぎないか
5. 次にGOLD / JPY / XYZ100などへ広げる価値があるか
6. RegimeGuardへ接続する価値があるか
7. Live Execution Monitorと比較する段階へ進めるか
```

つまり、これは **「勝てるBot」ではなく、「勝てる可能性のある売買ルールを現実的にふるいにかける道具」** です。
