# 結論

確認しました。
**指摘は正しいです。混線リスクはまだ残っています。**

ただし、これは

```text
live order permission が誤って出る危険
```

ではなく、

```text
人間・LLM・後続実装が before-cost proxy を actual cash と誤読する危険
```

です。

現行コードは gate ではかなり守っています。けれど、**表示名・schema名・report集計名がまだ危ない**。
ここは利益検証の根幹なので、次の最優先PRは `source_audit` や `fill_model` ではなく、まず **cash metric semantic repair** です。

# 確認結果

## 1. preview 経路は before-cost proxy を `actual_cash_result_usd` に入れている

`build_tournament_rows_preview()` では、matured outcome の `short_return_before_cost` / `long_return_before_cost` に notional を掛けて `reversal_cash` / `continuation_cash` を作っています。

その値を `TournamentEventResult(actual_cash_result_usd=...)` に詰めています。

一方で、known gaps には

```text
OUTCOME_BEFORE_COST_PROXY_NOT_ACTUAL_CASH
FEES_FUNDING_AND_FILL_SLIPPAGE_NOT_INCLUDED
```

が追加されています。

つまり、**artifactの意味はproxyだが、field名はactual cash** です。ここが一番危険です。

---

## 2. v2 rows はかなり安全に分離されている

`CostAwareTournamentRow` では、

```text
before_cost_proxy_usd
fee_estimate_usd
funding_estimate_usd
slippage_estimate_usd
operator_time_cost_usd
cost_adjusted_cash_estimate_usd
stress_cash_estimate_usd
evidence_level
actual_cash_result_usd
```

が分かれています。

さらに `CryptoPerpTournamentRowsV2` の primary metric は `cost_adjusted_cash_estimate_usd` になっています。

ここは良いです。
問題は、legacy/report 側の `TournamentEventResult` と `CryptoPerpTournamentReport` がまだ `actual_cash_result_usd` 前提である点です。

---

## 3. tournament report は actual cash 前提の名前で集計する

`CryptoPerpTournamentReport` は `primary_metric="actual_cash_result_usd"` 固定です。

また、score計算でも `row.actual_cash_result_usd` を集計しています。

なので、preview rows を report に渡した場合、known gaps があっても、表面上は actual cash metric に見えます。

---

## 4. gate は proxy を止めるようになっている

`tournament_gate.py` は proxy gap を見ています。

```text
OUTCOME_BEFORE_COST_PROXY_NOT_ACTUAL_CASH
FEES_FUNDING_AND_FILL_SLIPPAGE_NOT_INCLUDED
```

を `PROXY_KNOWN_GAPS` として定義しています。

そして proxy gap があれば、

```text
NEEDS_ACTUAL_CASH
REBUILD_WITH_ACTUAL_CASH
```

へ落とします。

したがって、**gate安全性はかなり守られている**。
ただし、reportやmarkdownだけ読む人間にはまだ誤読リスクが残る。

---

# 判断

現状を3段階で評価します。

```text
live permission risk:
  低い

gate bypass risk:
  低〜中

human / LLM misread risk:
  高い
```

だから直すべき順番はこれです。

```text
1. 表示・schema・reportの意味修正
2. legacy preview rows の隔離
3. actual cash source がない report を gate 以前に明示的に proxy 扱い
4. その後に source audit / fill model / OFI 自動計算
```

いきなり source audit や fill model に進むと、**名前が嘘をついたまま精密化する**ことになります。

# 実装内容

## 1. `TournamentEventResult` に basis を追加

既存互換を壊しすぎないため、まずはfield追加で対応します。

```python
cash_metric_basis: Literal[
    "before_cost_proxy",
    "cost_adjusted_estimate",
    "actual_cash",
] = "actual_cash"
```

さらに、

```python
cash_metric_label: str
```

または summary 側に以下を追加します。

```text
cash_metric_basis
cash_metric_display_name
```

### 重要

既存の `actual_cash_result_usd` はすぐ削らない。
ただし、**proxyを入れる場合は `cash_metric_basis="before_cost_proxy"` を必須**にする。

---

## 2. preview rows では basis を必ず proxy にする

`build_tournament_rows_preview()` の `TournamentEventResult` 生成時に以下を入れる。

```python
cash_metric_basis="before_cost_proxy"
```

NO_TRADE も同じevent setの比較値なので、

```python
cash_metric_basis="before_cost_proxy"
```

に統一する。

---

## 3. tournament report に basis を伝播する

`CryptoPerpTournamentReport` に追加。

```python
cash_metric_basis: Literal[
    "before_cost_proxy",
    "cost_adjusted_estimate",
    "actual_cash",
    "mixed",
]
primary_metric_display_name: str
```

集計時のルール：

```text
全row basisが actual_cash:
  cash_metric_basis=actual_cash
  primary_metric_display_name=actual_cash_result_usd

全row basisが before_cost_proxy:
  cash_metric_basis=before_cost_proxy
  primary_metric_display_name=before_cost_proxy_usd

全row basisが cost_adjusted_estimate:
  cash_metric_basis=cost_adjusted_estimate
  primary_metric_display_name=cost_adjusted_cash_estimate_usd

混在:
  cash_metric_basis=mixed
  tournament_status=INCONCLUSIVE_DATA
```

---

## 4. report markdown / stdout で actual cash と表示しない

`crypto-perp-tournament-report` の出力やmarkdownで、proxyの場合にこう出す。

```text
primary_metric=before_cost_proxy_usd
cash_metric_basis=before_cost_proxy
actual_cash=false
```

estimateの場合：

```text
primary_metric=cost_adjusted_cash_estimate_usd
cash_metric_basis=cost_adjusted_estimate
actual_cash=false
```

actualの場合だけ：

```text
primary_metric=actual_cash_result_usd
cash_metric_basis=actual_cash
actual_cash=true
```

---

## 5. gate は known_gaps だけでなく basis でも止める

現状は proxy known gaps によって止まります。これは残す。
加えて、report の `cash_metric_basis != actual_cash` なら、明示的に止める。

```text
before_cost_proxy:
  NEEDS_ACTUAL_CASH

cost_adjusted_estimate:
  READY_FOR_TINY_LIVE_SHADOW
  or NEEDS_ACTUAL_CASH
  ただし production / scale は不可

mixed:
  INCONCLUSIVE_DATA
```

このほうが、known gaps がどこかで落ちても安全です。
