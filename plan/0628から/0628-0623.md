結論。
**いまのRepoに対しては、もう“Profit-readiness layerを作る”ではなく、“Profit-readiness layerの誤読・手入力・推定値依存を潰す”段階です。**

追加調査で見えた最大の問題はこれです。

> **コード上は profit-readiness まで進んでいるが、まだ「推定利益」と「実現現金」が混ざりやすい構造が残っている。
> ここを潰さないと、かなり綺麗に負けるシステムになる。**

現行Repoには、`crypto-perp-source-availability`、`crypto-perp-replay-slice`、`crypto-perp-feature-pack`、`crypto-perp-edge-score`、`crypto-perp-tournament-rows-v2`、`crypto-perp-bias-guard`、`crypto-perp-tiny-live-shadow` がすでに local artifact として使える状態です。

なので、次は新規構築ではなく、**利益に直結しない甘さを削るPR群**にします。

# まず疑うべき誤謬

## 1. `actual_cash_result_usd` がまだ危ない

ドキュメントでは `actual_cash_result_usd` は実fill・実fee・実funding・cash ledgerに接続している時だけ使う、と明記されています。

しかし、現行 `tournament_rows_preview` は before-cost の値を `actual_cash_result_usd` に入れています。known gapで「proxy not actual cash」と残してはいますが、field名はまだ危険です。
さらに `tournament.py` 側は `primary_metric="actual_cash_result_usd"` 固定で、score集計もそのfieldを使います。

これは、かなり実務上まずいです。

**修正方針：**

```text
旧:
  before-cost proxy row -> actual_cash_result_usd fieldへ入る

新:
  before_cost_proxy_usd
  cost_adjusted_cash_estimate_usd
  stress_cash_estimate_usd
  actual_cash_result_usd は actual source がある時だけ
```

`known_gaps`で注意するだけでは足りません。人間もLLMも後で誤読します。

---

## 2. `source_availability` は「ある/ない」で、まだ「使える/使えない」ではない

`source_availability.py` は `can_compute_ofi = can_compute_depth` としています。
しかし、booksが存在するだけではOFIは使えません。

必要なのはこれです。

```text
books exists
ではなく
books are usable
```

具体的には、

```text
coverage ratio
max gap ms
cutoff respected
timestamp monotonic
sequence monotonic
checksum valid
symbol一致
row count > 0
```

が必要です。

現状は source refs とCLI入力への依存が大きく、`--available-source` や `--row-count` で補う設計です。

**修正方針：**

`source_availability` の上に `source_audit` を追加する。

---

## 3. feature pack はまだ手入力寄り

`CryptoPerpFeaturePack` は `trade_sign_imbalance`、`ofi`、`depth_10bps` を持っています。
しかしCLIでは `--trade-sign-imbalance`、`--ofi`、`--depth-10bps` として外から渡す形です。

つまり、いまはまだ、

```text
trades / books artifact
  -> 自動で特徴量化
```

ではありません。

これは利益に近いところで非常に重要です。
出来高よりOFIやBBO depthを見た方が短期価格変化に強いという研究があります。特にCont, Kukanov, Stoikovの研究では、短時間の価格変化は単純な出来高よりorder flow imbalanceと関係が強く、その影響はmarket depthに反比例すると報告されています。([arXiv][1])

だから、今のRepoで次に金に近いのは、**OFIを語ることではなく、trades/books artifactから自動計算すること**です。

---

## 4. edge scorer はまだ粗い

`edge_rules.py` は、現状では event return、spread、funding だけで score を作っています。

これは初期MVPとしては良いですが、利益判断としては弱い。

今のルールだと、以下が十分に入っていません。

```text
trade sign imbalance
OFI
depth_10bps
depth_25bps
spread deterioration
mark-index basis
book/trade coverage
actual fillability
```

**修正方針：**

edge scorerをML化しない。
ただし、deterministic ruleの入力を増やす。

```text
edge_scorer_v2:
  source_audit
  feature_pack
  fill_sim
  funding_basis_check
```

を読ませる。

---

## 5. bias guard はサンプル不足を“失敗”扱いしすぎ

現行 `bias_guards.py` では、`pbo_status=NOT_ESTIMABLE` の時に `sample_sufficient_for_pbo` がfailし、stop reasonになります。

これは「strategy adoption」を止めるなら正しい。
しかし「実データdogfood」や「tiny-live shadowへ進む準備」まで全部止めると、学習速度が落ちます。

**分けるべきです。**

```text
MEASUREMENT_READINESS:
  PBO_NOT_ESTIMABLEでも進める場合がある
  ただし known_gap として明示

STRATEGY_ADOPTION:
  PBO_NOT_ESTIMABLEなら止める
```

現状はこの2つが混ざっています。

---

## 6. tiny-live shadow と credential readiness に意味のズレがある

`account.py` の `measurement_readiness_blockers()` は、`trade_enabled` がtrueだと `CREDENTIAL_NOT_READ_ONLY` でblockします。
これはread-only probeとしては正しい。

一方、将来のtiny-live measurementを準備するなら、trade-enabled keyが必要になる場面があります。
`tiny_live_shadow.py` では trade_enabled の場合に `TRADE_ENABLED_ATTESTED_BUT_NOT_USED_BY_SHADOW` を known gapへ入れています。

ここは誤読リスクがあります。

**修正方針：**

```text
READ_ONLY_ACCOUNT_PROBE
  trade_enabled=false 必須

TINY_LIVE_SHADOW_PREFLIGHT
  trade_enabled=trueでもよい
  ただし exchange_write_used=false
  live_order_submitted=false
  withdrawal_disabled_confirmed=true
  ip_restriction_confirmed=true
```

この2つを別policyに分ける。

# 追加調査で本当に見るべきテーマ

教科書的な「もっと高度なモデル」ではなく、次です。

## 調査1: Fillabilityを潰す

追加研究・OSS調査対象：

```text
hftbacktestのfill model
Tardis Bitget books/trades sample
Bitget WS books1/books15仕様
```

hftbacktestは、market replayでは自分の注文が板に影響しないという制約があり、fill simulationには楽観/悲観の罠があると説明しています。つまり、完璧なsimulationではなく、**どれだけ嘘をついているかを知る道具**として使うべきです。
また、queue positionやlatencyの重要性もhftbacktestの中心テーマです。

調査の目的はこれ。

```text
100 USD / 250 USD が本当に食えるか
best / mid / worst fill bandでまだ残るか
books欠損時にNO_TRADEへ落とせるか
```

---

## 調査2: Sequential stopping

個人は、長期研究よりも「負け仮説を早く殺す」方が重要です。

オンライン実験では、何度も途中結果を見ると誤検出が増えるため、逐次検定やfutility stoppingが使われます。最近のSPRT-z研究も、固定期間テストに繰り返し覗き見をするとType I errorが増える問題と、早期停止の実務価値を扱っています。([arXiv][2])

MarketLensへ転用するなら、p値ではなくこうです。

```text
event familyを続けるか
打ち切るか
tiny-live shadowへ進めるか
```

使う指標：

```text
累積stress_cash_estimate
NO_TRADEとの差分
最大損失
operator時間
sample不足
```

**やること：**

```text
event family stop rule
```

を作る。

例：

```text
20 matured eventsで、
NO_TRADEとの差分がマイナス
かつ stress_cash_estimate < 0
なら REJECT_OR_REVISE
```

## 調査3: Security / SRE的な“死因ランキング”

利益のために一番実務的なのは、勝ちパターン探しより、候補の死因を集めることです。

```text
TRADE_GAP
FUNDING_BASIS_RISK

```

これを毎週集計する。

**目的は、勝った事例を見ることではなく、どこで金にならないかを知ること。**
