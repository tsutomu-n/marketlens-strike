# 1117_バックテストについて

## このノートの扱い

バックテストの基本注意点を、戦略採用前のゲートとして扱う。

## 元ノートの要旨

バックテストの考え方、注意点、検証に関する短いメモ。

## 今日時点での補正

バックテストは「儲かる証明」ではなく「捨てるための最初の検査」。良い結果より、どの条件で壊れるかを先に見る。

## 理想的ナラティブ / 誤謬リスク

- `BACKTEST_OVERFIT`: 成績の良い期間だけを見て採用する。
- `EXECUTION_GAP`: 実際の約定・手数料・遅延を無視する。
- `PREDICTION_OVERCLAIM`: 過去の勝ちを将来期待値と混同する。

## 戦略部品への分解

- `Evaluation Harness`: train/test、walk-forward、stress。
- `Cost Model`: fee、slippage、funding、borrow、taxは別管理。
- `Rejection Rules`: 採用条件より先に捨て条件。
- `Decision Log`: 試行回数と変更理由を記録。

## 実験に落とすなら

全候補に共通する最小バックテスト仕様を先に固定する。あとから都合よく評価方法を変えない。

## 採用条件

ウォークフォワード、コスト2倍、期間変更、銘柄変更の最低4条件で大きく崩れないこと。

## 捨て条件

パラメータ、期間、銘柄を少し変えると期待値が消える場合。

## 現在性チェック

使う取引所の手数料、funding、約定仕様、データ粒度を確認する。
