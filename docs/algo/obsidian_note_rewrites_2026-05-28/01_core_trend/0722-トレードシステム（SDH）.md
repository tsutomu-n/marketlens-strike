# 0722-トレードシステム（SDH）

## このノートの扱い

低レイテンシー/多機能トレード基盤構想としてではなく、データハブ、監視、実行、運用環境のリスク棚卸しとして使う。

## 元ノートの要旨

仮想通貨スキャルピング向けのSynergy Data Hub、低レイテンシー、VPS、Cloudflare、実装計画、機能一覧、簡易版、トレード戦略を含む。

## 今日時点での補正

SDHは構想が大きく、最初に作ると保守負担が重い。まずは「データ鮮度を監視する小さいハブ」「実行前の安全ゲート」「ログと停止理由」の3点だけに削るべき。

## 理想的ナラティブ / 誤謬リスク

- `OPERATIONAL_COMPLEXITY`: データ統合基盤があれば戦略が強くなるという前提。
- `MEV_LATENCY_ARMS_RACE`: VPSやCDNで低レイテンシー問題を解決できるという前提。
- `SECURITY_SECRET`: APIキー保管や接続設定が増え、漏洩面が広がる。

## 戦略部品への分解

- `Data Collector`: market data, order book, on-chain, newsの取得。
- `Monitoring Layer`: freshness、latency、error rate、drop rate。
- `Execution Adapter`: 取引所別の最小実行層。
- `Risk Guard`: kill switch、資金上限、API停止。

## 実験に落とすなら

最初は取引しない。価格データ、板、API応答、シグナル候補を1週間記録し、欠損率、遅延、異常停止の頻度を見る。

## 採用条件

ハブ導入によって、戦略検証の再現性と障害把握が明確に改善すること。

## 捨て条件

低レイテンシーや多機能化が主目的になり、期待値検証より先にインフラ作業が膨らむ場合。

## 現在性チェック

VPS/取引所リージョン、Cloudflareの役割、取引所WebSocket仕様、API制限は現在の公式情報で確認する。
