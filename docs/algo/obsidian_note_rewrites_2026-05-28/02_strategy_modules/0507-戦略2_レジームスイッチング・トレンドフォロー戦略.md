# 0507-戦略2 レジームスイッチング・トレンドフォロー戦略

## このノートの扱い

トレンド戦略を常時稼働させず、相場状態ごとに参加/停止/縮小を切り替える設計候補。

## 元ノートの要旨

レジームを判定し、トレンドフォローの稼働条件やパラメータを切り替える。

## 今日時点での補正

レジーム判定は未来を当てるものではなく、誤判定しても損失が限定される「運転モード切替」として使う。複雑なHMMやMLより、まずはボラ、トレンド強度、出来高、スプレッドでよい。

## 理想的ナラティブ / 誤謬リスク

- `PREDICTION_OVERCLAIM`: レジームを正しく識別できる前提。
- `BACKTEST_OVERFIT`: レジーム閾値を過去に合わせやすい。
- `OPERATIONAL_COMPLEXITY`: モード切替が多すぎて挙動を説明できなくなる。

## 戦略部品への分解

- `Regime Detector`: trend, range, high-vol, low-liquidity。
- `Participation Filter`: 稼働/停止/縮小。
- `Position Sizer`: レジーム別の最大リスク。
- `Risk Guard`: レジーム急変時の強制停止。

## 実験に落とすなら

常時トレンドフォローと、レジームフィルタ付きトレンドフォローを比較する。改善指標は総利益よりも、レンジ期間の損失削減と高ボラ急変時のDD削減。

## 採用条件

見送りによる機会損失より、避けた損失の方が安定して大きいこと。

## 捨て条件

レジーム判定が遅く、トレンド終盤で参加してレンジで損する場合。

## 現在性チェック

レジーム判定に使うデータの取得遅延と欠損を確認する。
