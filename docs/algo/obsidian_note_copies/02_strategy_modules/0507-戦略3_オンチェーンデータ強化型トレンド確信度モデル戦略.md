**戦略3: オンチェーンデータ強化型トレンド確信度モデル戦略**

**コンセプト:**
標準的なテクニカル指標によるトレンドシグナルをベースとしつつ、仮想通貨特有のオンチェーンデータを活用してそのトレンドの「質」や「持続可能性」を評価する「確信度スコア」を算出し、取引判断の精度向上とリスク管理の強化を目指す戦略。

**OSSからの着想点:**

*   **多様なデータソースの可能性:** `the_virtue_of_complexity_everywhere.ipynb` は、多数の（潜在的に非線形な）特徴量を用いることで予測モデルのパフォーマンスが向上する可能性を示唆しています。オンチェーンデータは、価格データとは異なる次元の情報を提供し、これに合致するアプローチです。
*   **ファンダメンタルズとテクニカルの融合:** 伝統的市場でも、テクニカル分析にファンダメンタルズな要素を加味することで、より堅牢な投資判断ができるという考え方があります。オンチェーンデータは、仮想通貨における「ファンダメンタルズ」の一部と見なすことができます。
*   （間接的に）`commodity_crack_spread_stat_arb.ipynb` や `commodity_crush_spread_stat_arb.ipynb` で見られるような統計的分析や回帰モデルは、複数の要因を組み合わせて市場を理解しようとする試みであり、本戦略の「確信度スコア」生成にも類似した発想が活かせます。

**詳細ロジック:**

**対象資産:**
オンチェーンデータが豊富かつ信頼性の高い主要な仮想通貨（例: BTC, ETH, および主要なL1/L2プロトコルトークン）。

**時間軸:**
日足データを基本とし、オンチェーンデータも日次集計値を用いる。

**ステップ1: ベーストレンドシグナル生成**

1.  **指標:** 標準的で実績のあるテクニカル指標を用いる。
    *   **例1 (EMAクロス):** 50日EMAと200日EMAのクロス。
        *   50日EMA > 200日EMA → 上昇トレンド (+1)
        *   50日EMA < 200日EMA → 下降トレンド (-1)
    *   **例2 (MACD):** MACDラインとシグナルラインのクロス、またはMACDヒストグラムの符号。
2.  **シグナル:** 上記指標から、当日のベーストレンド方向（+1: 上昇、-1: 下降、0: ニュートラル）を決定。

**ステップ2: オンチェーンデータ収集と特徴量エンジニアリング**

以下のオンチェーンデータを収集し、トレンド確信度評価のための特徴量に加工する。データソースはGlassnode, CryptoQuant, Dune Analytics, The Graphなど、または自前ノード。

1.  **ネットワーク活動系:**
    *   **アクティブアドレス数:** 過去N日平均からの変化率、絶対水準。
    *   **新規アドレス数:** 過去N日平均からの変化率。
    *   **トランザクション数/量:** 過去N日平均からの変化率、高額トランザクションの割合。
2.  **ホルダー行動系:**
    *   **取引所への資金フロー (Netflow):** 大量のインフローは売り圧力、アウトフローは買い・長期保有を示唆。過去N日間の累積Netflow、1日のNetflowの標準偏差からの乖離。
    *   **クジラ (大口保有者) の行動:** クジラの総保有量の変化、クジラによる取引所への送金/引き出し。
    *   **HODL Waves / Realized Cap HODL Waves:** 長期保有者の割合の変化。
    *   **SOPR (Spent Output Profit Ratio):** 市場参加者の平均的な利益/損失確定状況。SOPR > 1 は利益確定、< 1 は損失確定。
3.  **マイニング/ステーキング系 (該当する場合):**
    *   **ハッシュレート/Difficulty (PoW):** ネットワークのセキュリティと参加者のコミットメント。急変は注意。
    *   **ステーキング量/比率 (PoS):** ネットワークへの参加度と売り圧力の低減。
4.  **評価指標系:**
    *   **NVT Ratio (Network Value to Transactions Ratio):** 時価総額をトランザクション量で割ったもの。高すぎる場合は割高を示唆する可能性。その変化率。
    *   **MVRV Ratio (Market Value to Realized Value Ratio):** 時価総額を実現時価総額で割ったもの。市場の過熱度や底値圏を示唆する可能性。

**特徴量エンジニアリング:**
生データだけでなく、変化率、移動平均、過去N日間の最大/最小、標準偏差からの乖離（Zスコア）などを計算し、特徴量として利用する。

**ステップ3: トレンド確信度スコア算出**

収集・加工したオンチェーン特徴量を用いて、ベーストレンドシグナルの「確信度」を評価するスコアを算出する。

1.  **ルールベース・スコアリング:**
    *   各オンチェーン特徴量に対して、トレンドを支持する方向（例：上昇トレンド時にアクティブアドレス増加ならプラス点）と、その影響度（重み）を設定。
    *   例:
        *   上昇トレンドシグナル (+1) の場合:
            *   アクティブアドレス増加率 > 閾値A → +0.2ポイント
            *   取引所からのNetflow < 閾値B (マイナスが大きい) → +0.3ポイント
            *   SOPR > 1 かつ上昇傾向 → +0.1ポイント
        *   各ポイントを合計し、正規化して確信度スコア（例: -1から+1の範囲）とする。
    *   このルールと重みは専門家の知見とバックテストによる調整が必要。

2.  **(オプション) 機械学習モデルによるスコアリング:**
    *   過去のベーストレンドシグナル発生時とその後のトレンド持続期間/リターンを目的変数とし、オンチェーン特徴量を説明変数として機械学習モデル（例: ロジスティック回帰、ランダムフォレスト、勾配ブースティング）を学習。
    *   学習済みモデルを用いて、現在のオンチェーンデータからトレンド持続確率や期待リターンを予測し、これを確信度スコアとする。

**ステップ4: 取引判断とポジションサイジング**

1.  **エントリー判断:**
    *   ステップ1のベーストレンドシグナルがオン（+1 または -1）。
    *   かつ、ステップ3のトレンド確信度スコアが、トレンド方向に対して一定の閾値以上（例: 上昇トレンドなら確信度スコア > 0.3）。
    *   上記両方を満たした場合にエントリー。

2.  **ポジションサイジング:**
    *   基本は戦略1と同様のボラティリティ・スケーリング（目標ボラティリティ / 実現ボラティリティ）。
    *   さらに、トレンド確信度スコアの高さに応じてポジションサイズを調整。スコアが高いほど基本サイズに近づけ、低いがエントリー閾値は超えている場合はサイズを縮小する（例: スコアに応じて0.5倍〜1.0倍の係数を乗じる）。

**ステップ5: イグジット**

*   損切り: ATRトレーリングストップなど、標準的なトレンドフォローの損切り手法。
*   利益確定:
    *   ベーストレンドシグナルの反転。
    *   トレンド確信度スコアが大幅に低下し、一定の閾値を下回った場合（オンチェーン状況が悪化）。
    *   固定RRR。
*   オンチェーンデータに基づくアラート（例：取引所への異常な大口流入など）を早期警戒シグナルとして利用することも検討。

**パラメータの例 (初期設定、ルールベーススコアリングの場合):**

*   ベーストレンド: 50日EMA vs 200日EMA
*   オンチェーンデータ特徴量:
    *   アクティブアドレス: 過去30日平均比
    *   取引所Netflow: 過去7日累積
    *   SOPR (14日MA)
*   確信度スコアルール: (詳細な閾値設定は省略、バックテストで決定)
*   エントリー閾値 (確信度スコア): 0.3
*   実現ボラティリティ: 21日 (Yang-Zhang)
*   目標ボラティリティ: 年率40%

**バックテストと最適化:**

*   オンチェーンデータの履歴取得と正確性の担保が重要。
*   各オンチェーン指標のトレンドへの影響度（重み）や閾値の最適化。
*   機械学習モデルを用いる場合は、特徴量選択、モデル選択、過学習対策（クロスバリデーションなど）が不可欠。
*   オンチェーンデータの「先行性」と「遅行性」を理解し、シグナル生成のタイミングを調整。

**留意点:**

*   **データコストと可用性:** 高品質なオンチェーンデータフィードは有料の場合が多く、コストがかかる。無料データソースは信頼性や更新頻度に注意。
*   **解釈の難しさ:** オンチェーン指標は多岐にわたり、その解釈が一筋縄ではいかない場合がある。複数の指標を組み合わせることでノイズを減らす努力が必要。
*   **市場構造の変化への対応:** 仮想通貨市場が成熟するにつれて、オンチェーンデータの示す意味合いも変化する可能性があるため、定期的なモデルの見直しが必要。

この戦略は、テクニカル分析の限界を補うために、よりファンダメンタルに近いオンチェーンデータを活用する試みです。OSSの `the_virtue_of_complexity_everywhere.ipynb` が示唆するように、多様な情報を適切に組み合わせることで、予測力の向上が期待できますが、その分、実装と維持の難易度は高まります。

def preprocess_onchain_data(raw_onchain_df, params):
    """ 生のオンチェーンデータを加工し、モデル用の特徴量を作成する """
    df = raw_onchain_df.copy()
    processed_features = pd.DataFrame(index=df.index)

    # 例1: アクティブアドレス数のN日変化率 (対数差分などの方が良い場合も)
    if 'active_addresses' in df.columns:
        processed_features['active_addresses_pct_change'] = df['active_addresses'].pct_change(params['onchain_feature_lookback'])
        # Zスコア化 (過去N日の平均と標準偏差で標準化)
        mean_aa = processed_features['active_addresses_pct_change'].rolling(window=params['onchain_zscore_lookback']).mean()
        std_aa = processed_features['active_addresses_pct_change'].rolling(window=params['onchain_zscore_lookback']).std()
        processed_features['active_addresses_zscore'] = (processed_features['active_addresses_pct_change'] - mean_aa) / std_aa.replace(0,np.nan)


    # 例2: 取引所のNetflow (例: BTC建て) のN日累積とZスコア
    if 'exchange_netflow_btc' in df.columns:
        processed_features['exchange_netflow_cumulative'] = df['exchange_netflow_btc'].rolling(window=params['onchain_feature_lookback']).sum()
        mean_nf = processed_features['exchange_netflow_cumulative'].rolling(window=params['onchain_zscore_lookback']).mean()
        std_nf = processed_features['exchange_netflow_cumulative'].rolling(window=params['onchain_zscore_lookback']).std()
        processed_features['exchange_netflow_zscore'] = (processed_features['exchange_netflow_cumulative'] - mean_nf) / std_nf.replace(0,np.nan)
        
    # 例3: SOPR (Spent Output Profit Ratio) のN日移動平均
    if 'sopr' in df.columns: # SOPRは既に比率なので、そのままか移動平均
        processed_features['sopr_ma'] = df['sopr'].rolling(window=14).mean() # 例: 14日MA
        # SOPRが1を上回っているか/下回っているか、その度合いも特徴量にしうる
        processed_features['sopr_above_1_strength'] = np.maximum(0, processed_features['sopr_ma'] - 1)
        processed_features['sopr_below_1_strength'] = np.maximum(0, 1 - processed_features['sopr_ma'])


    # ... 他のオンチェーン指標 (MVRV, NVT, HODL Wavesなど) も同様に処理 ...
    
    return processed_features.fillna(method='ffill').fillna(0) # 欠損値処理 (要検討)

def calculate_trend_confidence_score_rule_based(base_trend_signal, processed_onchain_features_row, params):
    """ ルールベースでオンチェーンデータから確信度スコアを算出 (より構造化) """
    score = 0.0
    # 各指標のウェイト (バックテストや統計分析で決定)
    weights = {
        'active_addresses_zscore': 0.3,
        'exchange_netflow_zscore': 0.4,
        'sopr_above_1_strength': 0.15,
        'sopr_below_1_strength': 0.15
    }
    
    # 上昇トレンド (+1) の場合の期待されるオンチェーンの動き
    if base_trend_signal == 1:
        # アクティブアドレス増 (Zスコア > 0.5 など)
        if processed_onchain_features_row.get('active_addresses_zscore', 0) > 0.5:
            score += weights['active_addresses_zscore']
        # 取引所からの資金流出 (Netflow Zスコア < -0.5 など)
        if processed_onchain_features_row.get('exchange_netflow_zscore', 0) < -0.5:
            score += weights['exchange_netflow_zscore']
        # SOPRが1を大きく上回る (利益確定が進んでいるが、まだ強気)
        if processed_onchain_features_row.get('sopr_above_1_strength', 0) > 0.05: # 例: SOPR MA > 1.05
            score += weights['sopr_above_1_strength']
            
    # 下降トレンド (-1) の場合の期待されるオンチェーンの動き
    elif base_trend_signal == -1:
        # アクティブアドレス減 (Zスコア < -0.5 など)
        if processed_onchain_features_row.get('active_addresses_zscore', 0) < -0.5:
            score -= weights['active_addresses_zscore'] # スコアを負にする
        # 取引所への資金流入 (Netflow Zスコア > 0.5 など)
        if processed_onchain_features_row.get('exchange_netflow_zscore', 0) > 0.5:
            score -= weights['exchange_netflow_zscore']
        # SOPRが1を大きく下回る (損失確定が進んでいる)
        if processed_onchain_features_row.get('sopr_below_1_strength', 0) > 0.05: # 例: SOPR MA < 0.95
            score -= weights['sopr_below_1_strength']
            
    return np.clip(score, -1.0, 1.0) # スコアを-1から1の範囲に正規化
