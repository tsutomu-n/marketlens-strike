**戦略2: レジームスイッチング・トレンドフォロー戦略**

**コンセプト:**
市場環境（レジーム）を統計的に判定し、現在のレジームに最適化されたトレンドフォロー戦略のパラメータセットまたは戦略自体を動的に切り替えることで、様々な市場状況への適応力を高める戦略。

**OSSからの着想点:**

*   **多様なバックテスト結果:** OSS内の多数のNotebook（例：`trend_following_moskowitz2012.ipynb`, `commodity_momentum.ipynb`, `equity_etf_intraday_momentum.ipynb` など）は、特定の戦略が市場環境（例：危機時、平常時）によってパフォーマンスが大きく変動することを示唆しています。これは、市場レジームの存在と、それに合わせた戦略調整の重要性を示唆します。
*   `equity_etf_intraday_momentum.ipynb` でのローリングOLSや拡大OLSによる係数の時間変化分析は、市場の予測構造が時間とともに変化すること（＝レジームシフト）を示唆しています。
*   （直接的ではないが）`the_virtue_of_complexity_everywhere.ipynb` は、複雑な（非線形な）モデルが市場の異なる状態を捉えるのに役立つ可能性を示しています。レジーム判定自体が非線形な問題である可能性があります。

**詳細ロジック:**

**対象資産:**
流動性の高い主要な仮想通貨（例: BTC, ETHなど）。

**時間軸:**
日足データを基本とし、レジーム判定と戦略実行を行う。

**ステップ1: 市場レジームの定義と判定**

市場レジームを少なくとも2つ、できれば3つ定義する。

1.  **レジームの定義例:**
    *   **レジーム1: 高ボラティリティ・トレンド相場 (High-Vol Trend Regime):** 価格変動が大きく、明確な方向性（上昇または下降トレンド）が出やすい状態。
    *   **レジーム2: 低ボラティリティ・レンジ相場 (Low-Vol Range Regime):** 価格変動が比較的小さく、方向感に乏しい持ち合い状態。
    *   **(オプション) レジーム3: 極高ボラティリティ・クラッシュ相場 (Crash Regime):** 極端にボラティリティが上昇し、急激な価格下落が見られるパニック的な状態。

2.  **レジーム判定指標の選定:**
    *   **ボラティリティ指標:**
        *   過去N日間の実現ボラティリティ（例：`realised_volatility.ipynb`で検討されたYang-Zhang推定器）。
        *   ATR (Average True Range)。
        *   仮想通貨ボラティリティ指数（例：DeribitのDVOL、BitMEXの.BVOLなど、もしあれば）。
    *   **トレンド強度指標:**
        *   ADX (Average Directional Index)。
        *   移動平均線の傾きや収束度合い。
    *   **(オプション) 市場心理指標:**
        *   Fear & Greed Index (仮想通貨版)。
        *   SNSセンチメントスコア。

3.  **レジーム判定ロジック:**
    *   **隠れマルコフモデル (HMM):**
        1.  上記の判定指標（またはその組み合わせ）を観測変数とし、市場がいくつかの隠れた状態（レジーム）間を遷移すると仮定してHMMを学習させる。
        2.  学習済みモデルを用いて、各時点において最も確率の高いレジームを判定する。
    *   **閾値ベースのルール:**
        1.  例えば、「実現ボラティリティが過去100日間の75パーセンタイル以上かつADXが25以上」→ レジーム1。
        2.  「実現ボラティリティが過去100日間の25パーセンタイル以下かつADXが20以下」→ レジーム2。
        3.  上記以外、または特定の急落パターン（例：1日で-15%以上の下落かつボラティリティ急騰）→ レジーム3。
        *   このルールはバックテストを通じて最適化が必要。

**ステップ2: レジーム別トレンドフォロー戦略の定義**

各レジームに対して、異なるパラメータセットを持つトレンドフォロー戦略、または異なるタイプのトレンドフォロー戦略を用意する。

1.  **レジーム1 (高ボラティリティ・トレンド相場) 用戦略:**
    *   **タイプ:** 比較的短期のトレンドを捉えることを重視。
    *   **シグナル例:** 短期EMAクロスオーバー（例：10日EMA vs 30日EMA）、短期チャネルブレイクアウト（例：20日間高値/安値）。
    *   **損切り/利益確定:** ATRベースのトレーリングストップ。ボラティリティが高いため、ATRの倍率はやや大きめ（例：3倍）に設定し、トレンドを追いやすくする。
    *   **ポジションサイジング:** ボラティリティ・スケーリングは行うが、トレンド発生時の初期段階ではやや積極的なサイジングも許容。

2.  **レジーム2 (低ボラティリティ・レンジ相場) 用戦略:**
    *   **タイプ:** トレンドフォロー戦略のシグナル発生頻度を抑えるか、レンジ相場に強い戦略（例：オシレーター系の逆張り、ただしこれはトレンドフォローの範疇から外れるため注意）に切り替える。ここではトレンドフォローに絞るなら以下。
    *   **シグナル例:** 長期EMAクロスオーバー（例：50日EMA vs 150日EMA）、長期チャネルブレイクアウト（例：100日間高値/安値）。シグナル発生の閾値を厳しくする（例：ADXが30以上など）。
    *   **損切り/利益確定:** レンジ相場ではトレンドが持続しにくいため、利益確定は早め（例：固定RRR 1:1.5）、損切りもタイトに。
    *   **ポジションサイジング:** ポジションサイズを全体的に小さくするか、取引を見送る。
    *   **(オプション):** このレジームではトレンドフォローは実行せず、待機する。

3.  **(オプション) レジーム3 (極高ボラティリティ・クラッシュ相場) 用戦略:**
    *   **タイプ:** リスク回避を最優先。
    *   **シグナル例:** 全てのロングポジションを決済。場合によっては、短期的な急反発を狙った非常に短期のショート戦略（高度なリスク管理が必要）。
    *   **ポジションサイジング:** 最小限、またはゼロ。

**ステップ3: 戦略の実行とリバランス**

1.  **日次処理:**
    1.  当日の終値データ等を用いて、ステップ1のロジックで現在の市場レジームを判定。
    2.  判定されたレジームに対応するトレンドフォロー戦略（ステップ2で定義）を適用。
    3.  その戦略に基づいて、翌日のエントリー/イグジットシグナル、および目標ポジションサイズを決定（ボラティリティ・スケーリングは各レジーム戦略内で適用）。

2.  **エントリー/イグジット:**
    *   翌日の始値でシグナルに基づき取引を実行。
    *   レジームが切り替わった場合、既存のポジションが新レジームの戦略と矛盾する場合は決済し、新レジームの戦略に従って新たにポジションを構築する。

**パラメータの例 (初期設定、HMMを使わない場合):**

*   **レジーム判定:**
    *   RV期間: 20日 (Yang-Zhang)
    *   ADX期間: 14日
    *   レジーム1: RV > 75パーセンタイル かつ ADX > 25
    *   レジーム2: RV < 25パーセンタイル かつ ADX < 20
*   **レジーム1戦略:**
    *   EMAクロス: 10日 vs 30日
    *   ATR: 14日, ストップ倍率 3.0
    *   目標ボラティリティ: 年率50%
*   **レジーム2戦略:**
    *   EMAクロス: 50日 vs 150日 (かつADX > 30を条件に追加)
    *   ATR: 14日, ストップ倍率 2.0, RRR 1:1.5
    *   目標ボラティリティ: 年率20% (または取引なし)

**バックテストと最適化:**

*   レジーム判定ロジックの精度が戦略全体のパフォーマンスに大きく影響するため、HMMの学習や閾値ベースのルールの最適化が非常に重要。
*   各レジームにおけるトレンドフォロー戦略のパラメータも個別に最適化。
*   レジームの誤判定リスクや、レジーム遷移時の取引コスト（スリッページ含む）も考慮。
*   異なる市場期間（ブル、ベア、レンジ）での堅牢性を確認。

**留意点:**

*   **レジーム判定の遅延:** HMMや移動平均ベースの指標は、どうしてもレジームの変化を後追いで判定するため、実際の変化から少し遅れて戦略が切り替わる可能性がある。
*   **過学習リスク:** レジームの定義や判定ロジック、各レジームの戦略パラメータが過去データに過剰適合するリスクがある。アウトオブサンプルテストが不可欠。
*   **計算負荷:** HMMを用いる場合、特にオンラインでの学習や判定は計算負荷が高くなる可能性がある。

このレジームスイッチング戦略は、市場の非定常性に対応しようとする高度なアプローチです。OSSでは明確にこの形での戦略は提示されていませんが、市場環境によって戦略パフォーマンスが変動するという普遍的な観察と、機械学習の応用（`the_virtue_of_complexity_everywhere.ipynb`）の可能性から着想を得ています。成功すれば、より安定した長期的なリターンが期待できるでしょう。

def determine_market_regime_threshold_based(price_df_history, params):
    """ 閾値ベースで市場レジームを判定する """
    df = price_df_history.copy()
    
    # 最新のデータポイントで評価するため、shift(1) を使って前日までのデータで閾値を計算
    df['rv_for_regime'] = calculate_rolling_std_volatility(df['Close'], params['rv_period'])
    df['adx_for_regime'], _, _ = calculate_adx(df['High'], df['Low'], df['Close'], params['adx_period'])

    # shift(1) を使って、当日の判定には前日までのデータに基づく閾値を使用
    df['rv_high_thresh'] = df['rv_for_regime'].rolling(window=params['rv_quantile_lookback'], min_periods=params['rv_quantile_lookback']//2).quantile(params['rv_high_q']).shift(1)
    df['rv_low_thresh'] = df['rv_for_regime'].rolling(window=params['rv_quantile_lookback'], min_periods=params['rv_quantile_lookback']//2).quantile(params['rv_low_q']).shift(1)

    current_rv = df['rv_for_regime'].iloc[-1]
    current_adx = df['adx_for_regime'].iloc[-1]
    rv_high = df['rv_high_thresh'].iloc[-1]
    rv_low = df['rv_low_thresh'].iloc[-1]
    
    # 欠損値処理 (初期データ不足の場合など)
    if pd.isna(current_rv) or pd.isna(current_adx) or pd.isna(rv_high) or pd.isna(rv_low):
        return "Undefined" # データ不足

    daily_return = df['Close'].pct_change().iloc[-1]
    rv_crash_thresh = df['rv_for_regime'].rolling(window=params['rv_quantile_lookback']).quantile(0.95).shift(1).iloc[-1]
    if daily_return < -0.15 and current_rv > rv_crash_thresh :
         return "Crash"

    # 主要レジーム判定
    if current_rv > rv_high:
        if current_adx > params['adx_trend_thresh']:
            return "HighVol_Trend"
        else:
            return "HighVol_Range" # 高ボラで方向性がない場合
    elif current_rv < rv_low:
        if current_adx < params['adx_range_thresh']:
            return "LowVol_Range"
        else:
            return "LowVol_WeakTrend" # 低ボラで少し方向性がある場合
    else: # 中間のボラティリティ
        if current_adx > params['adx_trend_thresh']:
            return "MidVol_Trend"
        else:
            return "MidVol_Range"

def apply_regime_hysteresis(regime_series, min_duration):
    """ レジームの頻繁な切り替えを防ぐヒステリシス処理 """
    processed_regime = regime_series.copy()
    last_regime = None
    last_regime_start_idx = -1
    for i in range(len(regime_series)):
        if last_regime is None or regime_series.iloc[i] != last_regime:
            # レジームが変わった場合、前のレジームが最小期間続いていたか確認
            if last_regime is not None and (i - last_regime_start_idx) < min_duration:
                # 続いていなければ、前のレジームで上書き
                processed_regime.iloc[last_regime_start_idx:i] = last_regime
            # 新しいレジームの開始
            last_regime = regime_series.iloc[i]
            last_regime_start_idx = i
    # 最後に残ったレジームもチェック
    if last_regime is not None and (len(regime_series) - last_regime_start_idx) < min_duration:
         processed_regime.iloc[last_regime_start_idx:] = regime_series.iloc[last_regime_start_idx-1] if last_regime_start_idx > 0 else last_regime # 最後のレジームが短すぎたら前のレジームに戻す

    return processed_regime


def run_regime_switching_strategy(full_price_df, regime_params_dict, 
                                  params_r_hvt, params_r_lvr, params_r_crash,

    df['raw_regime'] = df.rolling(window=max(regime_params_dict['rv_quantile_lookback'],1)+1).apply( # windowは適当に設定
        lambda x_df: determine_market_regime_threshold_based(x_df, regime_params_dict), raw=False
    ).fillna("Undefined")
    
    df['final_regime'] = apply_regime_hysteresis(df['raw_regime'], regime_params_dict['regime_hysteresis_days'])

    df['target_leverage_rs'] = 0.0
    df['atr_exit_rs'] = np.nan
    df['adx_exit_warning_rs'] = False

    # 2. レジームごとに対応する戦略ロジックを適用
    for regime_type, trend_params in [
        ("HighVol_Trend", params_r_hvt),
        ("MidVol_Trend", params_r_hvt), # MidVol_TrendもHighVol_Trendと同じ戦略を使う例

        if not regime_indices.empty:
            # trend_logic_function は戦略1の generate_trading_signals_and_positions に相当

            df.loc[regime_indices, 'atr_exit_rs'] = signals_for_regime['atr_exit']
            df.loc[regime_indices, 'adx_exit_warning_rs'] = signals_for_regime['adx_exit_warning']

    # "Undefined" や未定義のRange系レジームは target_leverage_rs = 0 のまま (取引なし)
    
    return df[['final_regime', 'target_leverage_rs', 'atr_exit_rs', 'adx_exit_warning_rs']]

**この改善版スニペットのポイント:**

1.  **レジーム判定の具体化と堅牢性向上:**
    *   閾値ベースのレジーム判定ロジックをより具体的にし、クラッシュレジームの判定例も追加しました。
    *   判定に使う閾値（RVのパーセンタイルなど）を、判定対象日の前日までのデータで計算するように修正しました（`shift(1)`）。これにより将来情報のリークを防ぎます。
    *   `apply_regime_hysteresis` 関数を導入し、レジームの短期間での頻繁な切り替え（チャタリング）を防ぐロジックを追加しました。これは実運用での安定性に寄与します。
2.  **レジーム別戦略のモジュール化:**
    *   `trend_logic_function` という引数を導入し、戦略1で定義したトレンドフォローのメインロジック関数を各レジームで異なるパラメータセット（`params_r_hvt`など）で再利用できるようにしました。これにより、コードの重複を避け、メンテナンス性が向上します。
    *   各レジーム（HighVol_Trend, LowVol_Range, Crashなど）に対応する具体的なパラメータセットを定義し、それぞれの市場環境に合わせた戦略調整のイメージを明確にしました。
3.  **パラメータ管理の改善:**
    *   レジーム判定用パラメータと、各レジームのトレンド戦略用パラメータを明確に分離し、辞書型で管理するようにしました。
4.  **HMMの導入可能性への言及:** コメントで `hmmlearn` ライブラリの使用を示唆し、より高度なレジーム判定手法への道筋を示しました。
5.  **網羅性の向上:** 以前はコメントアウトされていたレジーム別戦略の呼び出し部分を、具体的なループ処理と `loc` を使った結果の代入に置き換え、全体の流れを明確にしました。
6.  **欠損値処理の考慮:** RV計算初期や閾値計算初期の `NaN` によりレジーム判定ができないケースを考慮し、"Undefined" レジームを返すようにしました。また、`rolling().quantile()` に `min_periods` を設定して初期のデータ不足に対応する例を示しました。

**依然として残る課題・簡略化されている点:**

*   **HMMの完全実装:** HMMの学習と予測は依然として概念的な記述です。
*   **レジーム別戦略の最適化:** 各レジームでどのパラメータが最適かは、詳細なバックテストと分析が必要です。
*   **レジーム遷移時のコスト/遅延:** 擬似コードでは考慮されていません。
*   **`trend_logic_function`のインターフェース:** 戦略1の関数が日々のシグナルを返す前提ですが、レジームスイッチング戦略全体で過去データ全体に対して一度に計算しているため、呼び出し方に工夫が必要か、日次ループ内で逐次処理する形に合わせる必要があります（現在のスニペットは全期間一括計算の形）。

このスニペットは、戦略2のコアロジックをより具体的で実践的なものに近づけ、特にレジーム判定の堅牢性とレジーム別戦略の適用方法を明確にすることを目指しました。
