# 戦略アイディアの作り方 やさしい説明

## まず結論

この仕組みは、勝てる投資戦略を自動で見つけるものではありません。

やっていることはもっと地味です。

たくさんの戦略案を作り、危ない案や根拠が弱い案を落とし、残した案だけを次の検証へ回します。その時に、「どの案を作ったか」「なぜ落としたか」「どのデータを見たか」を全部残します。

一言で言うと、これは戦略案のふるい分け台帳です。

## 何のためにあるのか

投資戦略は、適当にたくさん試すと、偶然よく見える案が必ず出ます。

たとえば、100個の案を試して、その中で一番成績がよかった1個だけを見せると、すごく良い戦略に見えることがあります。でも実際には、ただの偶然かもしれません。

この repo の仕組みは、そこをごまかさないためにあります。

よさそうな案だけを残すのではなく、試した案、落とした案、残した理由、見たデータを残します。あとから見て「これは本当に検証に進めてよいのか」を判断できるようにします。

## 何が出てくるのか

出てくるものは、大きく3つです。

1. 戦略案の一覧
   作った案を全部並べたものです。残した案だけでなく、落とした案も入ります。

2. 落とした理由
   似た案だった、必要なデータが足りない、条件が危ない、未来の情報を見ている疑いがある、などを記録します。

3. 人間が読む確認資料
   何件作り、何件落とし、何件を次へ回したかを読むための資料です。

ここで出てくる戦略案は、まだ検証前です。実際に使える戦略ではありません。

## どういう流れか

流れは次の通りです。

1. 使うデータを確認する。
   データの出どころ、期間、更新時刻を見ます。

2. 決まった型で戦略案を作る。
   たとえば「流れに乗る案」「行き過ぎの戻りを見る案」「値動きの大きさを見る案」などです。

3. ダメな案を落とす。
   似すぎている案、材料不足の案、危ない条件の案を落とします。

4. 残した案だけを次へ回す。
   ただし、この時点でもまだ「使える戦略」ではありません。次の検証に進める候補です。

5. 全部を記録する。
   残した案だけでなく、落とした案も記録します。

## たとえ話

料理の新メニューを考える場面に近いです。

まず候補をたくさん出します。辛いメニュー、軽いメニュー、安く作れるメニュー、季節限定メニューなどです。

その後で、材料が手に入らないもの、原価が高すぎるもの、既存メニューとほぼ同じものを落とします。

最後に、試作してよい候補だけを残します。

この時点では、まだ人気メニューではありません。試作に進めてよい候補です。

この repo の戦略案も同じです。作っただけでは、まだ勝てる戦略ではありません。

## どんな戦略案を作るのか

主に次のような方向で案を作ります。

- 相場の流れが続くかを見る案。
- 上がりすぎ、下がりすぎの反動を見る案。
- 値動きが大きくなったか、小さくなったかを見る案。
- 売買しやすい状態かを見る案。
- 複数の銘柄を比べて、強いものや弱いものを見る案。
- 暗号資産先物では、資金調達率や価格差などを見る案。

これらは「勝ちやすい型」ではありません。検証前の案を作るための出発点です。

## この仕組みが特に気をつけていること

一番気をつけているのは、都合のよい見せ方をしないことです。

具体的には、次を避けます。

- 一番よく見えた案だけを見せる。
- 落とした案を隠す。
- 未来の情報を見たかもしれない案を通す。
- 過去データでよく見えただけで、勝てると呼ぶ。
- AIの案をそのまま採用する。
- 検証前の案を、実取引してよい案として扱う。

## 現在できること

今できるのは、戦略案を作り、整理し、次の検証へ渡すところまでです。

具体的には、次ができます。

- 戦略案の一覧を作る。
- 似た案や条件不足の案を落とす。
- 落とした理由を残す。
- 使ったデータの情報を残す。
- 人間が読む確認資料を作る。
- 残した案だけを次の検証へ渡す。
- 暗号資産先物向けの案を一定の型で作る。
- AIへ渡す材料を作り、人間が戻したAI回答を検証前の案として取り込む。

## まだできないこと

次はまだできません。

- 自動で儲かる戦略を発見する。
- 全ての案を自動で検証完了まで進める。
- 実際の取引コストを完全に測る。
- 紙上運用してよいと判断する。
- 実取引してよいと判断する。
- 取引所へ注文を出す。
- AIの提案をそのまま採用する。

## 読む時のポイント

この仕組みの結果を見る時は、成績の良い案だけを見ないでください。

次を順番に見ます。

1. 何件の案を作ったか。
2. 何件を落としたか。
3. なぜ落としたか。
4. 何件を次へ回したか。
5. 使ったデータは何か。
6. 未来の情報を見ていないか。
7. まだ検証前の案として扱われているか。

この順番で見ると、単に「良さそうな案」ではなく、「どういう探し方をして、どれだけ落として、なぜ残したか」が分かります。

## 1行でまとめると

この仕組みは、戦略案を作る道具というより、戦略案を雑に採用しないための道具です。

勝てそうに見える案を作ることより、後で検証できる形で残すこと、危ない案を落とすこと、検証前の案を実戦投入と勘違いしないことを重視しています。

## 用語の言い換え

元資料やコードには、英語の専門用語が出ます。この資料では次の意味で読めば十分です。

| 元の言葉 | ここでの意味 |
|---|---|
| Strategy Idea Candidate | 検証前の戦略案 |
| Strategy Idea Candidate Set | 作った戦略案の一覧 |
| candidate generation pipeline | 案を作って、落として、残す流れ |
| deterministic generator | 同じ条件なら同じ案を作る仕組み |
| Candidate Family | 案の型 |
| parameter grid | 条件の組み合わせ |
| rejection reason | 落とした理由 |
| input evidence | 使ったデータの情報 |
| leakage | 未来の情報をうっかり見てしまうこと |
| shortlist | 次の検証へ回す案だけに絞ること |
| backtest | 過去データで試すこと |
| paper / live permission | 紙上運用や実取引の許可 |
